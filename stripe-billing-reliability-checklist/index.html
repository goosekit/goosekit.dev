Stripe + Supabase billing reliability

A practical checklist for the billing bugs that quietly cost money.

A Stripe payment can succeed while the app state stays wrong. This checklist is for Next.js + Supabase SaaS builders who want to catch webhook, entitlement, dunning, portal, and reconciliation edge cases before users hit them.

Use this before launch, after launch, and after every billing change.

Do not test only the happy checkout path. Test the state transitions: what Stripe says, what Supabase stores, what the user sees, and how the app recovers when the first write fails.

Checklist

  • Store Stripe customer, subscription, price, and latest relevant event ids against the right user/team/workspace.
  • Verify webhooks with the raw body and endpoint secret in production, not only locally.
  • Make webhook handling idempotent with a processed-events table keyed by Stripe event id.
  • Do not unlock paid access from the success URL alone; treat redirect as UX, not source of truth.
  • Test duplicate, delayed, and out-of-order events with saved fixtures or Stripe CLI replays.
  • Run a recurring reconciliation scan for recent subscriptions, invoices, refunds, and portal changes.
  • Define what happens during trial end, past_due, unpaid, pause_collection, cancel_at_period_end, cancel, refund, and downgrade.
  • Log an exception queue: expected state, actual state, provider object id, local user/team id, detected time, resolution note.
