Stripe Connect + affiliate ledgers

Stripe Connect affiliate revenue share, but your earnings ledger can drift?

Affiliate revenue share looks like a growth feature, but it is also a money-state problem. One checkout can touch a connected account, creator earnings, platform fees, affiliate attribution, self-referral rules, refund reversals, and a monthly payout batch.

The useful review is narrow: one purchase, one attribution path, and one refund reversal should replay into the same local earnings ledger exactly once.

The invariant to protect

One paid event should produce one correct creator earning, one correct affiliate fee when attribution is valid, one platform fee, and one reversible money-state trail before any monthly payout batch runs.

Fixed-scope review fit: usually EUR149 when live creator earnings, refund reversals, self-referral blocking, or payout-batch replay are in scope.

1. Separate attribution from money movement

A referral cookie or code should not be the source of truth for money. It is an input. The final local ledger should show which creator earned money, which affiliate earned money, what the platform kept, and why.

Attribution truth

  • Referral code
  • Click or attribution row
  • Cookie window start and end
  • Self-referral block result

Money truth

  • Stripe Checkout or payment event
  • Connected account id
  • Creator earnings ledger row
  • Platform and affiliate fee config

2. Replay the refund path before payout day

The highest-risk bug often appears after the first happy-path sale. If a purchase is refunded after affiliate attribution was recorded, the local reversal needs to unwind creator and affiliate earnings before a payout batch pays stale balances.

  • The refund webhook reverses creator earnings but not affiliate earnings.
  • A duplicate refund event subtracts twice from the same affiliate balance.
  • The payout batch reads available earnings while a refund reversal is only half-written.
  • Self-referral blocking happens at click time but not at payout time.

3. Store one replay-safe earnings result

For a Stripe Connect platform, a good local model usually has a durable event key, signed ledger deltas, and a payout-batch boundary that only reads settled local truth.

  • Use provider event ids and local transaction ids as replay keys.
  • Record signed deltas rather than overwriting balances silently.
  • Return the same ledger result on duplicate purchase or refund delivery.
  • Do not include unsettled or disputed earnings in the payout batch.

4. Test these cases before trusting the launch

  • Valid affiliate purchase inside the cookie window.
  • Purchase with expired attribution.
  • Self-referral attempt.
  • Duplicate checkout or payment event delivery.
  • Refund after creator and affiliate earnings were credited.
  • Duplicate refund event delivery.
  • Monthly payout batch after a partial refund-reversal failure.
