Stripe webhook ordering + Supabase

Stripe webhooks arriving out of order?

If customer.subscription.updated, invoice.paid, or checkout.session.completed lands in a surprising order, your Supabase state can show the wrong plan, miss the user mapping, or overwrite an active subscription with stale data.

The production failure: “Stripe says active, my app says incomplete”

Do not design a webhook handler that only works when events arrive in the order you expect from a happy-path test. Subscription flows are asynchronous. Related events can be delivered close together, retried, or processed by your app in a different order than your mental model.

1. Identify the ordering bug you actually have

Most “out of order” bugs are one of these four shapes.

Updated before created

customer.subscription.updated writes active, then a later-processed created event overwrites the row back to incomplete.

Invoice before mapping

invoice.paid arrives before your app has stored which Supabase user/org owns the Stripe customer or subscription.

Checkout owns too much

checkout.session.completed both creates mapping and grants access, while subscription events also grant access with slightly different data.

Retry after partial failure

Your handler mutates Supabase, times out or returns 500, then a retry or adjacent event replays an older state transition.

  • Log sanitized event.id, event.type, event.created, customer, subscription, invoice, and internal user/org ID.
  • Compare Stripe's event created timestamp with your server received_at and processed_at timestamps.
  • Check whether a lower-fidelity event overwrites a row that a higher-fidelity event already corrected.
  • Separate “delivery order” from “processing order”: concurrent serverless invocations can finish in a different order than they started.

2. Store raw event facts before deciding business state

A small event ledger gives you enough information to replay, inspect, and avoid stale writes. This is also the foundation for duplicate delivery handling.

create table public.stripe_events (
  id text primary key,
  type text not null,
  stripe_created_at timestamptz not null,
  livemode boolean not null,
  customer_id text,
  subscription_id text,
  invoice_id text,
  received_at timestamptz not null default now(),
  processed_at timestamptz,
  processing_error text
);
await supabaseAdmin.from('stripe_events').insert({
  id: event.id,
  type: event.type,
  stripe_created_at: new Date(event.created * 1000).toISOString(),
  livemode: event.livemode,
  customer_id: getCustomerId(event),
  subscription_id: getSubscriptionId(event),
  invoice_id: getInvoiceId(event)
});

If this insert hits a unique violation on id, acknowledge the duplicate and do not run side effects again.

3. Make subscription writes monotonic

The core fix is to stop older or lower-confidence events from overwriting newer state. In Supabase/Postgres, keep Stripe timestamps and compare before writing.

  • Add last_stripe_event_created_at and last_stripe_event_id to the local subscription row.
  • Only overwrite status when the incoming event is newer than the row's last processed Stripe timestamp, or when the event is the chosen authority for that field.
  • Use stripe_subscription_id as the conflict key, not user email or a temporary Checkout Session ID.
  • Fetch the current Subscription from Stripe when a webhook payload is missing fields needed for a safe decision.
await supabaseAdmin.rpc('apply_stripe_subscription_event', {
  p_subscription_id: stripeSubscriptionId,
  p_customer_id: stripeCustomerId,
  p_status: stripeStatus,
  p_current_period_end: currentPeriodEnd,
  p_event_id: event.id,
  p_event_created_at: new Date(event.created * 1000).toISOString()
});
-- Inside the database function, guard against stale overwrites:
where subscriptions.last_stripe_event_created_at is null
   or subscriptions.last_stripe_event_created_at <= p_event_created_at;

4. Decide which event owns each job

Out-of-order bugs get worse when every event tries to do every job. Use events as signals, not as duplicate business workflows.

Checkout Session

Good for linking a checkout attempt to an internal user/org through trusted client_reference_id or metadata. Do not make it the only source for long-term subscription status.

Customer Subscription

Good for subscription status, plan, cancellation, pause, trial, and quantity changes. Treat it as the ongoing state owner for subscriptions.

Invoice Paid

Good for recording successful billing periods, extending entitlements, or clearing dunning state. It should attach to an existing customer/subscription mapping or create a pending repair item.

Customer Portal events

Good for changes after the initial checkout flow. Make sure portal updates cannot be overwritten by an older checkout-created path.

5. Add a pending state instead of dropping unmapped events

If invoice.paid arrives before your Supabase mapping exists, do not return success and lose it. Store it as pending, then resolve after checkout.session.completed or a Stripe API lookup fills the mapping.

create table public.stripe_pending_events (
  event_id text primary key,
  event_type text not null,
  customer_id text,
  subscription_id text,
  reason text not null,
  payload jsonb not null,
  created_at timestamptz not null default now(),
  resolved_at timestamptz
);
  • Do not silently skip invoice.paid because user_id is missing.
  • Do not key mapping by email; use Stripe customer/subscription IDs and trusted metadata captured when Checkout starts.
  • Do not let customer.subscription.created overwrite a newer updated status just because it is the “create” event.
  • Do not send customer-facing emails or grant credits until the mapping and dedupe checks pass.

What to send for an event-ordering review

Do not send secrets. Send the event sequence and schema facts that prove whether this is stale overwrite, missing mapping, duplicate processing, or an event-ownership problem.
