Vercel preview / production URL drift

Supabase Auth or Stripe Checkout keeps returning to the wrong Vercel URL?

If login works locally but production returns to localhost, a preview deployment, or an old Vercel domain — and Stripe success/cancel pages do the same — debug the base URL contract first. In this stack, Supabase Auth, Stripe Checkout, Vercel system envs, and Next.js server/client envs can each point at a different domain.

Treat auth callbacks and checkout returns as one URL system

The same root bug often breaks both flows: app code builds redirect URLs from NEXT_PUBLIC_SITE_URL, VERCEL_URL, branch URLs, request headers, or hard-coded strings without deciding which value is allowed in production.

Need a URL/env sanity pass? Auth callback checklist

Founding setup/debug help starts at €99–149 when the bug is narrow enough to inspect safely from sanitized env names, route files, and dashboard screenshots.

1. Name the canonical production URL explicitly

Do not let production discover its public URL from whichever deployment host was used to build the app. Set one boring canonical variable and use it for customer-facing redirects.

  • Use a variable such as NEXT_PUBLIC_SITE_URL=https://example.com or APP_URL=https://example.com in Vercel Production.
  • Keep preview URLs separate. Preview domains are useful for testing, but they should not become the permanent callback or checkout return URL for real users.
  • Do not put a trailing slash in one place and no slash in another without normalizing; it creates subtle double-slash and path bugs.
  • Never build a Stripe success_url from a user-provided query param or untrusted header.

2. Allow Vercel previews in Supabase only on purpose

Supabase Auth requires the redirectTo URL to match the configured redirect URL list. For previews, Supabase supports wildcard redirect URLs, but broad wildcards should be used deliberately because they expand what Auth will accept.

Production

  • Site URL: https://example.com
  • Redirect URL: https://example.com/auth/callback
  • Password recovery URL if separate.
  • OAuth provider callback configured as Supabase expects.

Preview / local

  • http://localhost:3000/auth/callback for local development.
  • Specific Vercel preview pattern only if your team actually tests auth in previews.
  • Do not make the Supabase Site URL a preview domain unless previews are the intended default.

3. Generate Supabase redirectTo from the same helper every time

OAuth, magic links, signup confirmation, and password reset should not each invent their own URL. Use one helper that returns the correct base URL for the current environment, then append the callback path.

function getAppUrl() {
  const explicit = process.env.NEXT_PUBLIC_SITE_URL || process.env.APP_URL;
  const vercelUrl = process.env.VERCEL_URL ? `https://${process.env.VERCEL_URL}` : '';
  const fallback = 'http://localhost:3000';

  const raw = explicit || vercelUrl || fallback;
  return raw.replace(/\/$/, '');
}

const redirectTo = `${getAppUrl()}/auth/callback`;

await supabase.auth.signInWithOAuth({
  provider: 'google',
  options: { redirectTo },
});

For production, prefer the explicit canonical URL. VERCEL_URL is helpful for preview testing, but it can describe a deployment URL rather than the custom production domain customers expect.

4. Build Stripe success/cancel URLs server-side from the canonical URL

Stripe Checkout stores the exact return URLs you send when creating the session. If your server sends a preview URL, Stripe will return the buyer to that preview URL even if the buyer started on production.

const appUrl = getAppUrl();

const session = await stripe.checkout.sessions.create({
  mode: 'subscription',
  line_items: [{ price: process.env.STRIPE_PRICE_ID, quantity: 1 }],
  success_url: `${appUrl}/billing/success?session_id={CHECKOUT_SESSION_ID}`,
  cancel_url: `${appUrl}/billing?checkout=cancelled`,
  metadata: { supabase_user_id: user.id },
});
  • Use Stripe webhooks for paid access. The success page is not guaranteed to load after payment.
  • Include {CHECKOUT_SESSION_ID} literally if the success page needs to retrieve the session.
  • Check the Checkout Session in Stripe Dashboard to see the exact success_url and cancel_url that were created.

5. Debug the actual URL that each system sees

Do not guess from code. Log or inspect the concrete URL values at the exact moment the session or auth link is created.

For Supabase Auth

  • Final redirectTo passed to signInWithOAuth or OTP.
  • Supabase Site URL and redirect allowlist screenshot, with project IDs okay but secrets hidden.
  • The final browser URL after callback failure.

For Stripe Checkout

  • Created Checkout Session ID.
  • Session success_url / cancel_url values in Stripe.
  • Vercel Production vs Preview env variable values by name and target, with secret values removed.

What to send for a URL/env review

Send the URL helper, auth login action, callback route, checkout-session creation route, sanitized Vercel env target summary, Supabase URL configuration, and one failing Stripe Checkout Session or auth redirect example.

Quick diagnosis table

Auth always returns to localhost

The Supabase Site URL or your app URL env is still local, or no redirectTo is being passed for that auth flow.

Stripe returns to a preview deployment

The checkout session was created with a preview-derived base URL. Inspect the session URLs in Stripe and fix the server-side app URL helper.

Preview auth fails, production works

The preview URL is probably not in Supabase's redirect allowlist. Add a deliberate preview pattern or test auth only on production/staging.

Success page loads but paid state is wrong

That is a webhook or user-mapping problem, not just a return URL problem. Confirm webhook delivery and Stripe customer → Supabase owner mapping.
