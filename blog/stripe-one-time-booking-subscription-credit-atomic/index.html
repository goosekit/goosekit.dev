One-time bookings + subscription credits

Stripe one-time bookings and subscription credits can drift.

A product can sell one-time paid sessions while also granting subscription credits. The risky part is not the checkout button. It is proving that booking creation, credit use, and Stripe events converge before money is captured or access is granted.

The failure mode is a charged user with no booking, a booked session with no paid state, or a subscription-credit path that bypasses the fail-before-charge rule.

The invariant to protect

Do not charge unless the session can be reserved atomically, and do not consume subscription credits unless the same booking row and replay guard prove exactly one paid session exists.

Fixed-scope review fit: EUR149 when one-time paid sessions, subscription credits, booking rows, and webhook replay can disagree.

1. Keep the booking row as the local anchor

Payment truth

  • Checkout Session or PaymentIntent id
  • Payment status and webhook delivery status
  • Processed-event row for duplicate delivery
  • Refund/cancel path if reservation fails

Booking truth

  • Booking/session row created atomically
  • Session type, slot, teacher/provider, or resource id
  • Subscription credit balance before/after
  • User-visible booking confirmation state

2. Test the cross-path failure cases

  • One-time checkout succeeds but booking insert fails.
  • Booking row is reserved but payment fails or webhook never arrives.
  • Subscription credit is consumed and Stripe checkout also charges for the same session.
  • Duplicate webhook creates two sessions or consumes two credits.
  • Concurrent booking attempts reserve the same slot before payment state converges.

3. Use one replay matrix for paid sessions

checkout succeeds + booking insert fails -> no confirmed booking, refund/repair row exists
booking reserved + payment fails -> slot released, no credit consumed
subscription credit session -> one credit consumed, no checkout charge
duplicate webhook -> processed-event hit, no second booking
concurrent checkout -> one booking wins, the loser fails before charge

What to review before launch

Check checkout creation, slot reservation, webhook fulfillment, credit consumption, duplicate replay, and the user-facing booking confirmation. That keeps the scope small while covering the money-state bugs that become refunds.
