Stripe + Supabase paid-access trust debug

Stripe paid flag bypass in a Supabase app?

A dangerous shortcut in fast-built SaaS apps is trusting a client-supplied paid, isPro, plan, or tier value to unlock paid features. If the server accepts that flag as truth, a user can often bypass Stripe without a valid subscription.

The safer pattern is boring: Stripe webhooks write trusted entitlement state, Supabase policies and server routes read that state, and client flags are treated as display hints only.

The invariant to protect

Only server-trusted billing state should grant paid access. A browser request can ask for a feature; it should not be able to declare the user paid.

Run the billing drift check Need a narrow review?

Founding review scope: €99-149 when the paid-access path, webhook write, and Supabase entitlement check are narrow enough to inspect safely.

1. Find every source of paid truth

Search for names like isPro, is_paid, plan, tier, subscriptionStatus, and credits. Then separate display state from authorization state.

  • Client display state can show badges, upgrade prompts, and cached UI hints.
  • Server authorization state decides whether an API route runs or a paid feature unlocks.
  • Supabase Row Level Security should not rely on user-writable profile fields for paid access.
  • Webhook handlers should be the only path that upgrades trusted paid state after Stripe proof.

2. Replace client flags with a trusted entitlement row

A compact entitlement table makes the boundary easier to audit. The important part is not this exact schema; it is that normal users cannot write their own paid access.

-- Example shape only: adapt names to your app.
create table billing_entitlements (
  user_id uuid primary key references auth.users(id),
  stripe_customer_id text,
  stripe_subscription_id text,
  plan_key text not null,
  status text not null,
  current_period_end timestamptz,
  updated_from_stripe_at timestamptz not null default now()
);

-- Normal app users should not update this table directly.
  • Do not let a browser request submit isPro=true and persist it.
  • Do not trust a JWT claim that users can refresh from stale profile data after cancellation.
  • Do not unlock paid API routes from request body, query string, local storage, or client metadata.

3. Gate paid routes on the server

For each paid API route, load the authenticated user, read the trusted entitlement row with a server client, and fail closed when the paid state is missing, expired, or unknown.

// Shape, not drop-in code
const entitlement = await getTrustedEntitlement(user.id);

if (!entitlement || entitlement.status !== "active") {
  return new Response("Upgrade required", { status: 402 });
}

if (entitlement.currentPeriodEnd && entitlement.currentPeriodEnd < new Date()) {
  return new Response("Subscription expired", { status: 402 });
}

If you have team billing, the entitlement should usually belong to the workspace or tenant, not just the current user profile.

4. Regression tests worth keeping

  • A request body with isPro: true does not unlock the paid route.
  • A user-writable profile update cannot change the trusted plan.
  • A canceled Stripe subscription removes or downgrades server-trusted access.
  • A webhook write failure returns a retryable error or queues repair instead of silently granting access.
  • A failed entitlement write is not acknowledged as done; it leaves a retryable processed-event or repair record.
  • A checkout success page cannot write a Pro profile value that bypasses the trusted entitlement row.
  • A stale client cache can show the wrong badge, but the paid API still fails closed.

What to send for a paid-access review

Send facts, not secrets: the paid route path, where isPro/plan/tier is read, the Supabase table that stores access, and the Stripe webhook code that updates it.
