Stripe setup mode + card-on-file gates

Stripe setup mode card gate looks like subscription access?

Checkout mode: setup is useful when you need to collect a payment method before a free tier, trial, import, or backfill. It is not the same thing as a subscription, paid invoice, or durable entitlement.

The risky version is a card-on-file gate that unlocks local access or starts expensive work while the app has not stored enough SetupIntent, customer, account, and job state to repair the flow later.

The invariant to protect

A setup-mode Checkout Session, the resulting SetupIntent, the saved payment method, and the local account row must point to the same user or team before access, imports, backfills, or usage jobs start.

1. Do not treat a saved card as paid state

mode: setup creates a SetupIntent flow. It can prove that a payment method was collected. It does not prove that the customer has an active subscription, paid invoice, trial entitlement, usage allowance, or completed billing agreement beyond the setup flow you designed.

  • Local access flips to active after Checkout returns from a setup session.
  • A backfill, import, or usage job starts before the payment method is attached to the expected account.
  • The app stores the Checkout Session id but not the SetupIntent id or payment method id.
  • The later subscription/trial path cannot tell whether the card gate already ran.

2. Put account metadata where future webhooks can use it

Setup-mode Checkout has several objects in play. If the only useful account identifier lives on the Checkout Session, later SetupIntent or payment method flows may not have enough context for repair.

// Example shape only: adapt names to your app.
await stripe.checkout.sessions.create({
  mode: "setup",
  customer: stripeCustomerId,
  client_reference_id: accountId,
  metadata: { accountId },
  setup_intent_data: {
    metadata: { accountId }
  },
  success_url,
  cancel_url
});
  • Store the Checkout Session id, SetupIntent id, Stripe customer id, local account id, and payment method state.
  • Use the same account identifier in session metadata and SetupIntent metadata when the app depends on later SetupIntent events.
  • Keep the local row idempotent: one account should not create several active card-gate records for the same purpose.

3. Gate jobs separately from card collection

Card-on-file is often used before a costly operation: historical import, AI generation, enrichment, bulk processing, or usage-based billing. That job/backfill gate needs its own state machine instead of assuming “card collected” means “safe to run everything.”

Billing setup state

  • Checkout setup session created
  • SetupIntent succeeded
  • Payment method attached
  • Local card gate marked ready

Work state

  • Job requested
  • Billing setup verified
  • Job enqueued once
  • Failure/refund/repair path recorded

4. Keep the replay matrix small but real

  • Setup Checkout returns to the success URL before SetupIntent events are processed.
  • The success URL is refreshed twice.
  • The same account starts two setup sessions in two tabs.
  • SetupIntent succeeds, but the local DB write fails.
  • Payment method attaches to the customer, but the local account row is missing or belongs to another team.
  • The expensive job is enqueued once, retried once, and blocked when setup state is incomplete.

5. The minimum query to add

Adapt the table names, but keep the question: can one local account have more than one open card gate or queued job for the same billing setup purpose?

select
  account_id,
  purpose,
  count(*) as open_gate_count
from billing_setup_gates
where status in ('pending', 'ready')
group by account_id, purpose
having count(*) > 1;
  • Add the inverse check too: job queued with no ready card gate.
  • Log failed DB writes after SetupIntent success as repairable incidents, not just console errors.
  • Make the UI honest: “card saved” is different from “subscription active” or “job complete.”
