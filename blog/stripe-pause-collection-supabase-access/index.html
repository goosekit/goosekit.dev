Stripe paused billing + Supabase access

Stripe pause_collection but Supabase access still active?

pause_collection can stop or change invoice collection without magically knowing what your product should allow. If Supabase only checks subscription.status === 'active', paused billing can leave customers with the wrong access.

The production failure: “paused billing” is not a product-access policy

Stripe's pause-collection controls billing behavior. Your app still needs an explicit decision: keep access during a courtesy pause, switch to read-only, limit usage, or lock the account until billing resumes. That decision must be stored and tested in Supabase.

1. Separate Stripe pause settings from app access

A subscription can remain active while collection is paused. That means your app needs to inspect and store more than the top-level status.

Billing state

Subscription status, pause_collection behavior, resume date, invoice handling, latest invoice state, and whether collection was resumed manually.

Access state

Full access, courtesy access, read-only mode, usage-limited mode, locked access, or a custom support/admin state.

  • Store stripe_status and stripe_pause_collection separately.
  • Store access_state separately from billing state so support/admin choices are visible.
  • Record pause_started_at, pause_resumes_at, and pause_behavior when present.
  • Do not assume active means “billable and fully entitled” if pause collection exists.

2. Decide the product policy before implementing handlers

There is no universal answer. A B2B SaaS may keep access during a negotiated billing pause. A usage-heavy API may stop expensive features immediately. Make the policy explicit.

const accessPolicy = {
  active_no_pause: { access: 'full' },
  active_pause_void: { access: 'courtesy', showAdminNotice: true },
  active_pause_keep_as_draft: { access: 'limited', showBillingNotice: true },
  active_pause_mark_uncollectible: { access: 'limited', showBillingNotice: true },
  resumed: { access: 'full' },
  canceled: { access: 'locked' }
};

These labels are examples. The important bit is that paused billing and product access are intentionally mapped, rather than implied by a Stripe status string.

3. Sync the events that change pause state

Pause/resume bugs often happen because the app never stores the changed subscription snapshot, or an older webhook retry overwrites the restored state.

  • customer.subscription.updated: sync pause_collection, status, price, period dates, cancellation, and resume details.
  • invoice.created / invoice.finalized: understand whether invoices are being voided, drafted, or marked uncollectible under your pause policy.
  • invoice.paid: restore or confirm full access when billing has resumed and payment actually succeeded.
  • customer.subscription.deleted: lock or downgrade access if the paused customer ultimately cancels.
  • Fetch the current Stripe Subscription before final writes when resume timing or webhook retries could create stale state.

4. Supabase fields that make pause bugs debuggable

alter table public.subscriptions add column if not exists stripe_status text;
alter table public.subscriptions add column if not exists stripe_pause_collection jsonb;
alter table public.subscriptions add column if not exists access_state text default 'locked';
alter table public.subscriptions add column if not exists pause_started_at timestamptz;
alter table public.subscriptions add column if not exists pause_resumes_at timestamptz;
alter table public.subscriptions add column if not exists billing_notice text;
alter table public.subscriptions add column if not exists last_stripe_event_id text;
alter table public.subscriptions add column if not exists last_stripe_event_created_at timestamptz;
  • Use a server-only Supabase client for webhook writes; admin billing state should not depend on the end user's session.
  • Keep an event ledger so retries do not duplicate or reverse pause/resume side effects.
  • Compare event timestamps or fetch fresh from Stripe before writing access-critical state.
  • Expose a support-readable reason for limited access instead of a mysterious permission error.

5. Test pause and resume like a revenue path

  • Pause collection and confirm the app moves to the intended access state, not whatever active used to mean.
  • Resume collection and confirm access returns only when your billing policy says it should.
  • Confirm a paused subscription cannot accidentally bypass usage limits, seat limits, or paid feature gates.
  • Test old webhook retries after resume; they must not put the account back into paused/limited state.
  • Test support/admin copy so the customer sees a billing-state message, not a broken app.

What to send for a pause/access review

Do not send secrets. Send the current Stripe subscription snapshot, the recent event order, and the Supabase table shape that controls access.
