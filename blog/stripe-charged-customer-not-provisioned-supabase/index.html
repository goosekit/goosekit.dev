Stripe + Supabase charged-and-stranded debug

Stripe charged the customer, but Supabase did not provision access?

This is the charged-and-stranded failure: Stripe says the customer paid or the subscription is active, but your app never created the tenant, workspace, subscription row, entitlement row, credits, seats, or dashboard access.

Do not start by rewriting Checkout. Start by proving exactly where the paid state stopped moving from Stripe into your product.

The invariant to protect

If Stripe says the customer is paid or active, the app must either have local access state or a visible recovery state. A paid user should not land on an empty dashboard with no way for the system to detect and heal it.

Founding charged-and-stranded review scope: €99-149 when the checkout/add-on path and local access tables are narrow enough to review safely.

1. Name the Stripe object that proves payment happened

Do not debug from the UI first. Pick the Stripe object that says the customer should have access.

  • checkout.session.completed: useful for the first handoff from Checkout into app state.
  • payment_intent.succeeded: useful for one-time payments, credits, or deposits.
  • invoice.paid: useful for subscription activation and renewal recovery.
  • customer.subscription.updated: useful for active, trialing, past_due, canceled, portal-change, or add-on state.

2. List the local rows that must exist after payment

A charged customer can still be stranded if only one of these writes failed. Make the expected local state explicit.

Provisioning rows

  • User/customer mapping
  • Tenant, org, or client row
  • Workspace or dashboard row
  • Membership / user-to-workspace map

Access rows

  • Subscription row
  • Plan/tier/price mapping
  • Entitlement / service access rows
  • Credits, seats, or usage limits

3. Decide whether success route, webhook, or reconciliation owns provisioning

The most fragile design is one where the browser return path is the only place that provisions paid access. Tabs close, redirects fail, and webhooks may arrive before the local user or tenant exists.

  • If the success route provisions access, add a backstop that detects paid-but-unprovisioned accounts.
  • If the webhook provisions access, make it handle missing local rows without silently dropping the entitlement write.
  • If a reconcile job heals access, verify it can create missing maps and entitlements, not only log the mismatch.
  • If two paths can run, every provisioning write must be idempotent.

4. Add the reconciliation query before another user hits it

The exact SQL depends on your schema. The shape is simple: find Stripe-active records that do not have local access.

-- Example shape only: adapt table names to your app.
select
  s.stripe_customer_id,
  s.stripe_subscription_id,
  s.status,
  s.updated_at
from subscriptions s
left join service_entitlements e
  on e.subscription_id = s.id
where s.status in ('active', 'trialing')
  and e.id is null;
  • Run it after each billing deploy until the failure mode is gone.
  • Keep an equivalent query for local-premium/Stripe-inactive drift.
  • Log the expected row, actual missing row, provider object ID, and recovery outcome.

5. Regression tests worth keeping

  • Success route runs twice and provisions only one tenant/workspace.
  • Webhook arrives before tenant provisioning is complete and does not lose the entitlement.
  • Checkout succeeds but the browser never returns; reconciliation heals the paid account.
  • Partial provisioning reruns safely without duplicate maps or entitlement rows.
  • Stripe-active/local-unprovisioned query returns zero after the healing job.
  • The user sees “provisioning/recovering” rather than a blank paid dashboard while repair is in progress.

What to send for a narrow charged-and-stranded review

Send facts, not secrets: one Stripe object, the local tables that should have been created, the route/webhook/reconcile code paths, and what the paid user currently sees.
