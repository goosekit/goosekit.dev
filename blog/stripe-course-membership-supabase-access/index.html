Stripe course memberships + Supabase access

Stripe course membership paid but Supabase access drifted?

Course platforms often start with one lifetime course, then become a recurring academy: multiple course ids, all-access memberships, access-code grants, and Stripe subscription plans. That is exactly where a simple paid flag starts breaking.

The risky bug is not just "webhook failed." It is when the subscription row says active, the course catalog says locked, an access code says unlocked, or a Stripe price env var points at the wrong plan.

The invariant to protect

For every logged-in learner, course access should be explainable from one current membership subscription or one explicit per-course grant. Checkout success and browser state should not be the source of truth.

1. Separate membership access from course grants

A good course gate can answer why access is allowed. "Has active subscription" and "has per-course grant" are different reasons and should be visible in the data.

Membership truth

  • Stripe customer id
  • Subscription id and status
  • Price id mapped to a membership plan
  • Current period and cancellation state

Course truth

  • Course id on protected content
  • Per-course entitlement or grant row
  • Access-code or comp override
  • Resolved access reason shown in support logs

2. Replay each Stripe event into the same access result

  • Checkout completes before the webhook updates the subscription row.
  • `customer.subscription.updated` changes plan or period but course gates keep stale access.
  • Portal cancellation sets cancel-at-period-end, but the app removes access immediately.
  • A duplicate webhook overwrites an access-code grant or creates two membership rows.
  • A missing Stripe price env var sends checkout into a graceful error but leaves a half-created local row.

3. Test access with course ids, not just user roles

If content is scoped by `course_id`, test the exact protected route and module id. A global role can look correct while `/learn/[course]` still leaks or blocks content.

membership active + course_id=ai-eng -> allowed by membership
membership canceled + course_id=ai-eng + grant row -> allowed by grant
membership canceled + course_id=foundations + no grant -> blocked
subscription updated + annual price -> same access, new renewal window

What to review before launch

Check the subscription lifecycle, portal cancel/change states, price env vars, membership resolver, per-course override path, and one protected course route. That keeps the review narrow enough to finish before a paid learner hits the bug.
