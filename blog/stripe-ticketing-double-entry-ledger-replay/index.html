Stripe ticketing ledger reliability

Moving ticketing payments to a double-entry ledger?

Ticketing and booking apps often start with mutable columns like attendee paid amount, remaining balance, refunded flags, listing totals, and modifier aggregates. A double-entry ledger can fix that, but only if payment, refund, replay, payout, and derived-balance rules all converge on the same append-only money history.

Use this before replacing mutable money fields with transfer rows. The goal is not accounting advice; it is replay-safe billing state for Stripe-backed ticket sales, refunds, fees, and payout reporting.

The invariant to protect

Every external money event should create or confirm exactly one local transfer batch. Derived attendee balances, listing totals, refund availability, provider fees, and payout reports should be read from that ledger instead of edited in separate places.

1. Choose the event guard before the ledger shape

A better schema will not save a webhook handler that can run twice. Start with the idempotency record or processed-payment guard that wraps the entire transfer batch.

  • Store the Stripe event id, payment intent, checkout session, or processor object that owns the batch.
  • Insert the processed-payment guard and transfer rows in one transaction.
  • Make webhook replay return the existing result instead of appending a second credit.
  • Keep manual repair paths behind the same guard or a separately named adjustment event.

2. Map every mutable money column to a transfer view

Before deleting columns, write down which ledger query replaces each old field. If a field cannot be derived, it may be business metadata rather than money state.

Old mutable state

  • attendee.price_paid
  • attendee.remaining_balance
  • listing.refunded or refund flags
  • modifier money aggregates
  • provider fee and payout columns

Ledger-derived state

  • Attendee receivable transfers
  • Payment application transfers
  • Refund/reversal transfers
  • Adjustment transfers with reason codes
  • Fee and payout reconciliation rows

3. Keep payment, refund, and adjustment rows symmetric

Partial and repeat refunds are where ticketing ledgers drift. A refund should not delete or mutate the original sale; it should append a reversal or refund transfer tied back to the original batch.

  • If a refund flips a boolean and edits paid amount, historical reports can stop matching Stripe.
  • If an attendee transfer is reversed but modifier transfers are not, listing totals drift.
  • If provider fees and payouts are ignored, organizer payout reports can look right while platform margin is wrong.
  • If tombstoned attendees disappear from queries, ledger history can become unreconcilable.

4. Replay the migration with one real ticket

Pick one attendee/order and walk it through the old columns and new ledger. You want boring equality before you migrate all rows.

payment event -> processed_payments guard
  -> attendee receivable transfer
  -> payment application transfer
  -> modifier fee or adjustment transfer
  -> provider fee transfer
  -> payout reconciliation target
  -> derived attendee/listing balance view
  • Webhook replay does not double-credit the attendee or listing.
  • Partial refund appends refund rows and leaves original payment rows intact.
  • Repeat refund cannot exceed refundable balance.
  • Provider fee and payout reports reconcile to the same transfer set.
  • Deleted or hidden attendees remain represented in ledger history.

5. Regression tests worth keeping

  • Duplicate Stripe event id returns the existing processed-payment batch.
  • Payment handler crash before transfer insert leaves a repairable terminal state, not half-credit.
  • Payment handler crash after transfer insert does not append on retry.
  • Refund creates reversal rows and updates derived balances without mutating old rows.
  • Modifier totals and attendee balances derive from the same transfer rows.
  • Payout reconciliation reads the ledger, not a stale listing total.
