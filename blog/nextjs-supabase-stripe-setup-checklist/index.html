SaaS setup checklist

Next.js + Supabase + Stripe setup checklist.

Use this before your MVP loses a weekend to auth callbacks, Stripe webhooks, portal state, environment variables, or “it works locally” billing bugs.

The short version

If your product is not ready for users, do not spend weeks polishing infrastructure. But if people can sign up or pay, the auth + billing layer must be boringly reliable.

1. Supabase auth callback checklist

  • Create a server-side /auth/callback route that exchanges the code for a session.
  • Test email confirmation, password recovery, OAuth login, logout, and expired links separately.
  • Confirm all email template URLs point to your production domain, not localhost or a preview URL.
  • Handle the “session loading” state explicitly so protected content does not flash open.
  • Keep service-role/admin Supabase keys server-only. Never expose them to client components.

2. Protected routes and dashboard access

  • Decide which routes require only login and which require an active paid status.
  • Fail closed for dashboard, billing, settings, and account routes.
  • Redirect unauthenticated users to login with a safe return URL.
  • Add a visible empty/error state when session refresh fails instead of looping forever.
  • Test protected routes in a clean browser profile, not only with your dev account.

3. Stripe checkout and customer portal

  • Use one internal product/price mapping instead of hardcoding Stripe IDs across components.
  • Pass a stable user/customer reference into checkout metadata.
  • Set success and cancel URLs for production, preview, and local development intentionally.
  • Create a customer portal route for billing updates and cancellations.
  • Test checkout with a brand-new user, an existing free user, and an already-paid user.

4. Stripe webhook checklist

  • Verify webhook signatures using the raw request body, not parsed JSON.
  • Handle duplicate webhook delivery idempotently.
  • Store enough event/customer/subscription data to debug billing state later.
  • Listen for checkout completed, subscription updated, subscription deleted, invoice paid, and invoice payment failed.
  • Test local webhooks with the Stripe CLI and production webhooks with a real endpoint.

5. Model paid-user state inside your app

Do not make the whole app ask Stripe on every page. Your app needs a small internal truth layer.

  • Track user/customer ID, subscription ID, plan, status, current period, and last webhook time.
  • Define states like free, checkout_started, active, trialing, past_due, canceled, and test/admin.
  • Make feature gates read from your internal state, not raw Stripe event names.
  • Add a manual admin/debug view before you need it in production support.

6. Final deployment sanity checks

  • Environment variables differ between local, preview, and production. Check all three.
  • OAuth callback URLs must match the exact deployed domain.
  • Stripe webhook endpoint must use the live signing secret in production.
  • Supabase redirect allowlist must include your production and preview URLs intentionally.
  • Run one full flow: sign up → confirm email → checkout → return → access paid route → open billing portal → cancel/test downgrade.
