Stripe webhook idempotency + Supabase

Stripe webhook creating duplicate subscription rows?

If a single Checkout payment creates two subscription rows, sends two welcome emails, adds credits twice, or flips paid state back and forth, do not assume Stripe is broken. Webhooks are retryable and can arrive more than once. Your handler needs to be idempotent.

The production failure: “it only happened once, but it charged real users”

Stripe can retry events, send duplicate event deliveries, and deliver related events close together. If your handler blindly inserts on every checkout.session.completed, customer.subscription.created, or invoice.paid, Supabase will faithfully store duplicate business state unless the database prevents it.

1. Prove which duplicate you have

There are two common shapes, and they need different fixes.

Same Stripe event delivered twice

The event.id is identical. This is a webhook delivery retry / duplicate receipt. Store processed event IDs and skip if already handled.

Different Stripe events touch the same row

checkout.session.completed, customer.subscription.created, and invoice.paid may all reference the same customer/subscription. Use business unique keys and upserts.

  • Log the sanitized event.id, event.type, created, customer, subscription, and internal user/org ID.
  • Compare duplicate rows: do they share the same Stripe subscription ID or only the same user ID?
  • Check whether your handler returns 500/timeout after the database write. Stripe may retry a successful mutation if your response failed.
  • Look for two handlers processing the same event: local Stripe CLI forwarding plus production endpoint, old endpoint still enabled, or multiple Vercel deployments.

2. Add an event ledger before business writes

A simple stripe_events table gives your webhook a memory. The important part is that stripe_event_id is unique and inserted before or inside the same controlled flow as the business update.

create table public.stripe_events (
  id text primary key,
  type text not null,
  livemode boolean not null,
  received_at timestamptz not null default now(),
  processed_at timestamptz,
  error text
);
// After Stripe signature verification, before mutating subscription state
const { error: eventInsertError } = await supabaseAdmin
  .from('stripe_events')
  .insert({ id: event.id, type: event.type, livemode: event.livemode });

if (eventInsertError?.code === '23505') {
  // Duplicate event delivery: acknowledge without doing the side effect again.
  return NextResponse.json({ received: true, duplicate: true });
}
if (eventInsertError) throw eventInsertError;

// Continue with idempotent business update...

In Postgres, 23505 is a unique-violation error. Handle it deliberately; do not treat it like an unknown failure.

3. Make the subscription table unique on Stripe IDs

The event ledger catches exact duplicate deliveries. It does not protect you from separate event types that all describe the same subscription. Your business tables also need unique constraints.

  • stripe_customer_id should usually map to one internal user/org in a separate customer mapping table.
  • stripe_subscription_id should be unique in your subscriptions table when subscriptions are first-class rows.
  • If your product allows one active subscription per org, add a partial unique index for active/trialing statuses or enforce it in a transaction.
  • Use upsert on stripe_subscription_id instead of unconditional insert.
create unique index subscriptions_stripe_subscription_id_key
on public.subscriptions (stripe_subscription_id)
where stripe_subscription_id is not null;

// Supabase upsert using the Stripe subscription as the conflict key
await supabaseAdmin
  .from('subscriptions')
  .upsert({
    user_id,
    stripe_customer_id,
    stripe_subscription_id,
    status,
    current_period_end,
    updated_at: new Date().toISOString()
  }, { onConflict: 'stripe_subscription_id' });

4. Do not use Stripe API idempotency keys as your webhook dedupe layer

Stripe API idempotency keys are for safely retrying your outbound create/update requests to Stripe. Webhook dedupe is different: your app is receiving events and must remember which event.id values and business objects it has already processed.

  • Do not assume “Stripe already handles idempotency” for your database writes.
  • Do not dedupe by email address. Emails change, and email is not a safe business key.
  • Do not mark the event processed before the critical business update unless you also track failure and replay safely.
  • Do not send irreversible side effects, like emails or credit grants, before dedupe succeeds.

5. Pick one event as the owner for each side effect

Many duplicate bugs happen because multiple event types try to create the same local state. Decide which event owns each job.

Usually good owner

checkout.session.completed can link the Checkout session to an internal user/org if you passed trusted metadata or client_reference_id.

Usually good updater

customer.subscription.updated and customer.subscription.deleted are better for ongoing status, cancellation, pause, and plan changes.

Billing confirmation

invoice.paid can extend access, but should update the existing subscription/customer record, not create a second user subscription blindly.

One-time side effects

Welcome email, onboarding task, or credit grant should have their own unique key such as event.id or invoice.id.

What to send for a duplicate Stripe webhook review

Do not send secrets. Send the evidence that proves whether the duplicate is event-level, business-key-level, concurrency, or endpoint configuration.
