Stripe Checkout success route + webhook truth

Stripe Checkout success URL unlocks access before the webhook?

This is a subtle billing-state bug: the user returns from Checkout, your success route retrieves the Checkout Session, and local access is upgraded before the verified webhook path has proven the durable subscription state.

The success URL can be a good UX confirmation. It should not become a second provisioning authority unless it is idempotent, replay-safe, and reconciled against the same local subscription truth as the webhook.

The invariant to protect

A Checkout success redirect, a duplicate browser refresh, and a duplicate checkout.session.completed webhook must all converge to one local subscription/access state. No double period extension, no early unlock without repairability, no silent DB write failure.

1. Treat the success URL as UX first

The browser redirect can show “payment received” or fetch read-only status. The dangerous step is when it also writes plan='pro', status='active', credits, seats, or entitlement rows.

  • Success route refresh runs the same provisioning code twice.
  • Success route succeeds locally while the webhook later fails or maps to a different user.
  • Success route writes active access but has no Stripe event id or idempotency key.
  • Success route and webhook both extend the current period or reset usage quotas.

2. Draw the three paths that can touch access

Return path

  • success_url
  • Checkout Session retrieve
  • Local flash / status page
  • Optional read-only sync

Durable path

  • Verified Stripe webhook
  • Event id / subscription id
  • Subscription or entitlement row
  • Retry / repair behavior

If both paths write state, they need the same source-of-truth row, the same idempotency guard, and the same recovery story.

3. Keep a replay matrix before launch

  • Success URL visited once before webhook delivery.
  • Success URL refreshed twice by the same browser.
  • checkout.session.completed delivered twice.
  • Webhook DB write fails after Stripe payment succeeds.
  • Webhook arrives after the local user, team, or subscription row changed.
  • customer.subscription.updated and customer.subscription.deleted still downgrade or repair the same row.

4. When Checkout replaces mock grants

Another version of this bug appears when an app starts with mock grants, then turns on hosted Stripe Checkout for several purchase flows at once. A subscription plan, a one-time credit pack, and a paid gift may all return through the same success route even though they should write different durable effects.

  • Stripe env variables are missing, so production quietly stays in mock-grant mode.
  • Plan subscriptions use mode='subscription', while credit packs and gifts use mode='payment', but the return path treats them like the same entitlement.
  • The success route verifies payment_status but does not store a pending-to-fulfilled Checkout Session row before applying the effect.
  • A gift row is created before payment, or created twice after refresh/webhook replay.
  • Initial subscription credits are granted at Checkout while recurring top-ups are assumed without a verified invoice.paid path.

5. Add the duplicate-provisioning query

The schema differs by app, but the query shape is consistent: find one provider object that created more than one local active access record.

-- Example shape only: adapt table names to your app.
select
  stripe_subscription_id,
  count(*) as local_rows
from subscriptions
where status in ('active', 'trialing')
group by stripe_subscription_id
having count(*) > 1;
  • Keep the inverse query too: Stripe-active object with no local active entitlement.
  • Run both after changing Checkout, webhook, portal, or reconciliation code.
  • Log the event id, Checkout Session id, local subscription row, and user-visible state.

6. Safer success-route behavior

  • Show “payment received, provisioning” until the local entitlement row is confirmed.
  • Fetch current local subscription state without mutating it when possible.
  • If the route performs repair, write a repair record and require provider object idempotency.
  • Return clear support/debug evidence when Checkout is paid but local access is not ready.

If the purchase creates a hosted tenant or workspace, check the hosted-plan entitlement replay path too, because the success URL may look correct while tenant provisioning is still missing.

If the Checkout Session is only collecting a card, use the narrower setup-mode card gate checklist instead of treating the success URL like paid subscription proof.
