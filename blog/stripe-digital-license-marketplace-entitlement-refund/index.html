Stripe + digital marketplace entitlements

Stripe marketplace paid, delivered, unlocked, and refunded states can drift.

Digital marketplaces look simple when the happy path is “list -> pay -> deliver -> unlock”. The risky part is that Stripe payment truth, local license truth, delivery status, refund clawbacks, creator availability, and buyer access all change asynchronously.

A useful review follows one paid item through purchase, durable delivery, entitlement unlock, duplicate webhook replay, and refund reversal. If those paths do not converge, buyers can lose access after paying, keep access after refunds, or receive duplicate licenses.

The invariant to protect

One Stripe purchase should create one durable license, one delivery record, one buyer entitlement, and one reversible refund/clawback trail. Every duplicate delivery or browser refresh should return the same local result.

1. Separate listing truth, payment truth, and license truth

A published item is not a license. A paid Checkout Session is not a delivery record. A success page visit is not durable access. Keep those states separate enough that they can be replayed and compared.

Provider truth

  • Checkout Session or PaymentIntent id
  • Customer and buyer account mapping
  • Payment, refund, and dispute status
  • Webhook event id and delivery status

App truth

  • Marketplace item/listing id
  • License or entitlement row
  • Delivery/download row
  • Refund or clawback ledger row

2. Treat refunds as entitlement events

The dangerous bug often appears after the first refund. Stripe can reverse the money while the app keeps the license active, or the app can revoke access before the refund is final and strand a legitimate buyer.

  • The success URL unlocks the file before the verified webhook writes the license.
  • Duplicate checkout.session.completed delivery creates duplicate licenses or duplicate download grants.
  • A refund webhook updates the payment row but leaves the entitlement active.
  • A failed delivery keeps the charge but never creates a repairable delivery state.
  • A creator payout or affiliate share reads stale earnings before the refund clawback is durable.

3. Use replay keys across the whole marketplace path

For digital products, the replay key should cover the business result, not only the Stripe event row. A duplicate event should return the existing license and delivery result, not create another grant.

  • Store processed provider event ids, then also enforce one active license per buyer/item/payment path.
  • Write delivery as a durable state: pending, delivered, failed_repairable, refunded, or revoked.
  • Make refund handling idempotent and visible in the same table or ledger that gates access.
  • Never let the browser return page be the only place that unlocks purchased content.

4. Test these cases before trusting the marketplace

  • Valid purchase, webhook delivery, and download unlock.
  • Success page loads before webhook delivery.
  • Duplicate checkout/webhook delivery.
  • Delivery write fails after Stripe payment succeeds.
  • Refund after license has been delivered.
  • Duplicate refund event delivery.
  • Refund after creator payout, affiliate credit, or revenue-share calculation.
