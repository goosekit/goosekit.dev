Stripe → Supabase identity mapping debug

Stripe payment succeeds, but the webhook cannot find the Supabase user?

If Checkout says paid but your app cannot unlock the account, the failure is often not Stripe delivery. It is identity mapping: the webhook has a customer, subscription, or checkout.session, but your Supabase tables do not have a trusted way to map that object back to the right user or organization.

Debug the mapping path, not just the webhook status

A 200 webhook response can still write the wrong row, write no row, or depend on email matching that breaks when customers change email, use team billing, or start Checkout while logged out. Trace the identifier from the signed-in Supabase user to Checkout metadata, then to the webhook event, then to the database row.

Need a billing mapping pass? General webhook checklist

Founding setup/debug help starts at €99–149 when the repo, Stripe event, and Supabase table shape are narrow enough to review safely.

1. Decide the source of truth: user ID, organization ID, or customer table

Before changing code, decide what actually owns billing access. Solo apps can often map a Stripe customer to auth.users.id. Team apps usually need an organization_id, workspace ID, or membership table instead.

  • Do not rely only on customer email for durable billing access. Emails can change and can be duplicated across auth, billing, and team invites.
  • If subscriptions belong to teams, store the team/workspace identifier, not only the individual user who clicked Checkout.
  • If a user can buy multiple plans or products, model subscriptions as rows instead of a single boolean on the profile.
  • Keep a table such as stripe_customers or billing_customers that links Supabase owner IDs to Stripe customer IDs.

2. Pass the app identifier when creating Checkout

The safest moment to link Stripe to your app is when a logged-in user asks your server to create a Checkout Session. Put a non-secret internal ID in metadata and/or client_reference_id, then persist the resulting Stripe customer ID after the session completes.

const session = await stripe.checkout.sessions.create({
  mode: 'subscription',
  customer: existingStripeCustomerId || undefined,
  customer_email: existingStripeCustomerId ? undefined : user.email,
  client_reference_id: user.id,
  metadata: {
    supabase_user_id: user.id,
    organization_id: activeOrgId,
  },
  subscription_data: {
    metadata: {
      supabase_user_id: user.id,
      organization_id: activeOrgId,
    },
  },
  line_items: [{ price: process.env.STRIPE_PRICE_ID, quantity: 1 }],
  success_url: `${appUrl}/billing/success?session_id={CHECKOUT_SESSION_ID}`,
  cancel_url: `${appUrl}/billing`,
});

Use IDs that are safe to expose to Stripe metadata. Never put service-role keys, access tokens, or private user data in metadata.

3. Know which metadata appears on which Stripe object

A common bug: metadata is added to the Checkout Session, then the code expects the same metadata to appear on customer.subscription.created or invoice.paid. Stripe objects are related, but metadata does not magically appear everywhere unless you set it in the right nested field or fetch the related object.

  • checkout.session.completed has the Checkout Session metadata and client_reference_id.
  • Subscription events describe a Subscription object; use subscription_data.metadata at Checkout creation if you need metadata there.
  • For one-time payments, use payment_intent_data.metadata if downstream PaymentIntent/Charge events need app IDs.
  • When metadata is missing, retrieve the Checkout Session, Subscription, or Customer explicitly instead of guessing from email.

4. Persist customer mapping idempotently in Supabase

Handle webhooks as repeatable state sync, not as one-shot UI callbacks. Stripe can retry events and events can arrive in an order you did not expect, so writes should be idempotent and keyed by durable Stripe IDs.

Minimum useful tables

  • billing_customers: owner type, owner ID, Stripe customer ID.
  • subscriptions: Stripe subscription ID, customer ID, owner ID, status, price ID, period dates.
  • webhook_events: event ID, type, processed timestamp, optional error.

Failure symptoms

  • Checkout is paid, but the app profile remains free.
  • The webhook logs say “no user found for customer”.
  • A returning customer creates a second Stripe customer accidentally.
  • Team billing unlocks the buyer but not the workspace.

5. Backfill safely when a live payment already happened

If someone already paid and the mapping failed, do not rerun random webhooks blindly. Create a short recovery path: identify the Stripe customer/subscription, confirm the intended Supabase user or org, write the customer mapping, then replay or manually sync the subscription row once.

  • Verify the payer identity outside the webhook before granting access manually.
  • Record the Stripe event ID or admin action used for the recovery.
  • After backfill, make future Checkout sessions reuse the stored Stripe customer ID.
  • Add logging around missing metadata/customer mappings so the next failure is visible immediately.

What to send for a mapping review

Do not send secrets. Send the Checkout creation route, webhook handler, sanitized Stripe event payload, the Supabase tables that should change, and the exact row state before/after the failed payment.

Quick diagnosis table

Metadata is empty on subscription events

You probably set Checkout Session metadata only. Set subscription_data.metadata or retrieve the Checkout Session from the completed event.

Webhook has only a customer ID

Look up the customer in your billing_customers table. If it is missing, your Checkout completion path never persisted the mapping.

Access unlocks for the wrong account

Stop matching by email. Use Supabase user/org IDs captured server-side while the user is authenticated.

Returning users create new customers

Before creating Checkout, check whether the Supabase owner already has a Stripe customer ID and pass it as customer.
