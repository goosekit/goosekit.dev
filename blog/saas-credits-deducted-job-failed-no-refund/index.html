Metered credits + async job billing

SaaS credits deducted, but the async job failed?

This is a money-state bug: the user loses credits, but the product did not deliver the output, did not refund, and did not leave a repairable terminal state.

Do not start by changing prices. Start by proving whether the debit ledger, job/output row, cron/request success state, and user-visible terminal state all describe the same outcome.

The invariant to protect

Every metered operation should end as delivered, refunded/reversed, or repair-needed. A debit row without output, refund, or repair state is a revenue leak and a trust leak.

Run the billing drift check Map a revenue leak

Founding fixed-scope review: EUR99 for one contained debit helper, EUR149 when multiple metered operations or async recovery paths can leak credits or money.

1. Separate the debit ledger from the job state

A credit system needs two sources of evidence. The billing ledger says money or credits moved. The job/output row says whether the paid action delivered.

Debit evidence

  • Debit helper or cost logger
  • Credit ledger row
  • User balance before and after
  • Idempotency key or request ID

Delivery evidence

  • Job/output row
  • Async worker or cron/job run status
  • Persisted output ID or file
  • User-visible terminal state

2. Make terminal states explicit

Ambiguous states create false-success bugs. Your app should not treat a request as billable just because a handler returned 200, a cron finished, or a debit helper ran.

  • delivered: debit stands and output is visible to the user.
  • refunded or reversed: debit is undone because no value was delivered.
  • repair_needed: debit happened, delivery is unknown or failed, and an operator/retry can reconcile it.
  • failed_no_charge: job failed before a durable debit or the debit was rolled back in the same transaction.

3. Replay the cases that usually leak credits

  • Insufficient balance: the job must not run and no ledger row should debit credits.
  • Debit success + output delivered: one debit row, one output row, user sees the result.
  • Debit success + downstream failure: refund/reversal or repair-needed row is created.
  • Duplicate retry: the same request does not debit twice or double-refund.
  • Stale cron recovery: a background job can find stranded debits and move them to a terminal state.

4. Add the stranded-debit query before another user hits it

The exact schema will differ. The shape is: find debits where no delivered output, refund, reversal, or repair record exists.

-- Example shape only: adapt table names to your app.
select
  l.id as ledger_id,
  l.user_id,
  l.amount,
  l.request_id,
  l.created_at
from credit_ledger l
left join job_outputs o on o.request_id = l.request_id
left join credit_refunds r on r.ledger_id = l.id
left join repair_records rr on rr.ledger_id = l.id
where l.type = 'debit'
  and o.id is null
  and r.id is null
  and rr.id is null;
  • Run it after changing any debit helper, async worker, cron, or output persistence code.
  • Keep the query close to the incident runbook, not buried in an analytics dashboard.
  • Log expected terminal state, actual terminal state, and repair outcome for each row.
