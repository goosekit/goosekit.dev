Multi-provider subscriptions + Supabase

Stripe, ZaloPay, or RevenueCat subscription bridge drift?

A SaaS can start with Stripe, add a regional provider like ZaloPay, then scaffold RevenueCat for mobile. The hard part is not the checkout button. It is making every provider event converge on one Supabase subscription truth.

The failure mode is a paid user whose provider state is valid while subscriptions, entitlements, or feature gates disagree because each provider path wrote a slightly different shape.

The invariant to protect

One user should have one explainable active entitlement per product scope, regardless of whether the latest money event came from Stripe Checkout, ZaloPay, RevenueCat, a portal change, or a replayed webhook.

1. Normalize provider events before writing app access

Do not let each provider mutate app access directly. Translate every payment event into a small internal subscription event first, then write subscriptions and entitlements from that normalized event.

Provider truth

  • Stripe customer, subscription, invoice, or Checkout Session
  • ZaloPay app trans id, amount, status, and callback signature
  • RevenueCat customer id, entitlement id, renewal, or cancellation event
  • Provider event id and replay/idempotency key

App truth

  • Supabase subscriptions row with provider and provider object id
  • Entitlement gate output for the affected user
  • Plan, product, or workspace scope
  • Processed-event row proving duplicate delivery is safe

2. Test the cross-provider conflict cases

  • Stripe subscription is active while ZaloPay callback for the same user is pending or failed.
  • RevenueCat renewal extends mobile access but leaves the web subscriptions row expired.
  • A portal cancellation in Stripe removes access even though the user has a separate mobile entitlement.
  • Duplicate callback delivery creates two active local rows for one plan scope.
  • A provider migration changes the provider object id but not the app entitlement source of truth.

3. Use one replay matrix per provider pair

Start with the two providers most likely to overlap for one buyer. The matrix should prove which local row wins, what gets ignored, and which duplicate events return 2xx without rerunning side effects.

Stripe active + ZaloPay pending -> access from Stripe, no duplicate row
ZaloPay paid + Stripe canceled -> access from ZaloPay if same plan scope
RevenueCat renewal + web row expired -> access extends through normalized event
Stripe duplicate event -> processed-event hit, no second entitlement write
Provider migration -> old provider row closed, new provider row active

What to review before launch

Check provider-to-internal-event normalization, duplicate callback handling, provider migration rules, entitlement resolver output, and one protected paid route. That keeps the scope small while covering the money-state bugs that become support tickets.
