Stripe hosted plans + tenant provisioning

Stripe hosted plan paid but tenant entitlement drifted?

Hosted-plan billing has a wider blast radius than a simple paid flag. A paid Checkout Session may need to resolve the buyer, map the subscription, create or update a tenant, attach platform features, and write a durable entitlement row.

The risky bug is when Stripe truth, entitlement truth, and tenant provisioning truth do not move together. A customer can pay and still get no hosted plan, or a retry can provision the same tenant twice.

The invariant to protect

One paid hosted-plan purchase should create one entitlement, mapped to one Stripe customer/subscription, one account or tenant, and one provisioning job/result. Checkout success should not be the source of truth.

1. Split payment, entitlement, and provisioning state

A hosted plan has at least three truths. If they share one Boolean, debugging becomes guesswork after the first retry, failed write, or tenant setup error.

Provider truth

  • Checkout Session id
  • Stripe customer id
  • Subscription id and status
  • Price or hosted-plan id

App truth

  • Entitlement resolver output
  • Tenant or workspace id
  • Provisioning job/result row
  • Feature/plan access snapshot

The entitlement row should explain why a tenant has access. The provisioning row should explain whether the hosted plan was actually created or repaired.

2. Make the entitlement resolver replayable

The resolver should be able to run from Stripe event truth plus local account mapping and produce the same entitlement result every time.

-- Example shape only.
create unique index if not exists tenant_entitlements_provider_key
  on tenant_entitlements (provider, provider_subscription_id, plan_key)
  where provider_subscription_id is not null;
  • Store Stripe ids separately from tenant ids and user ids.
  • Keep price-to-plan mapping explicit and versioned.
  • Return the existing entitlement on duplicate delivery.
  • Record resolver errors without marking the event fully applied.

3. Treat tenant provisioning as a separate dispatch

Provisioning often touches platform tables, background jobs, permissions, billing limits, or integrations. That step can fail after payment and entitlement writes succeed.

  • Checkout or webhook writes an entitlement but tenant setup fails.
  • Webhook returns 200 after only the billing event row was inserted.
  • Retry sees the event id and skips the missing provisioning job.
  • Tenant plan changes but feature flags or platform limits stay stale.

4. Keep event rows non-terminal until dispatch finishes

A billing event ledger should distinguish received, resolved entitlement, provisioned tenant, failed dispatch, and repaired. A single received-event idempotency check is not enough.

-- Example states, not a required schema.
received -> entitlement_resolved -> tenant_provisioned
received -> entitlement_failed -> repair_needed
entitlement_resolved -> provisioning_failed -> repair_needed
  • Only mark the event complete after entitlement and provisioning side effects are durable.
  • Retry failed local dispatch without duplicating tenant or entitlement rows.
  • Expose a repair state support can query before a customer reports missing access.

5. Replay these cases before launch

  • Checkout completed returns to the browser, but access waits for the verified webhook path.
  • First subscription payment resolves one hosted-plan entitlement.
  • Duplicate webhook delivery returns the same entitlement and tenant result.
  • Provisioning fails after payment, then retry completes without a duplicate tenant.
  • Plan change updates entitlement and tenant limits atomically enough to avoid mixed access.
  • Subscription cancel, unpaid, or deleted state removes future hosted-plan access without deleting history.
