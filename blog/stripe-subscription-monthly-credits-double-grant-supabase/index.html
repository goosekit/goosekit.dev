Stripe subscriptions + monthly credit grants

Stripe subscription renewed but monthly credits drifted?

Recurring subscriptions often need to grant a monthly credit package, usage allowance, seat quota, or course/session balance. The dangerous bug is not only “did Stripe renew?” It is whether one renewal created exactly one durable local grant.

If invoice.paid, customer.subscription.updated, retries, and local grant functions disagree, paid users can be blocked, or duplicate webhooks can mint extra credits.

The invariant to protect

One paid subscription cycle should produce one local grant, tied to one subscription, one customer/account, one cycle key, and one billing event ledger state. Checkout success should never grant credits by itself.

1. Separate subscription truth from grant truth

The subscription mirror answers “what is Stripe saying about this customer?” The grant row answers “what did this paid cycle give the user inside the app?” Keep both rows visible and linked.

Subscription state

  • Stripe subscription id
  • Stripe customer id
  • Status and period dates
  • Latest event timestamp

Grant state

  • Local account/student id
  • Cycle key
  • Credit/session/usage amount
  • Expires or renews at

One can update without the other during a failure. That is why you need a billing event ledger or repair queue instead of assuming the webhook handler either fully worked or fully failed.

2. Give every cycle an idempotency key

Stripe event ids are useful for delivery idempotency. Monthly grants also need product-level idempotency: the same subscription and same paid cycle should not grant twice just because Stripe delivered two events or your handler retried.

-- Example shape only: adapt table names to your app.
create unique index if not exists credit_grants_cycle_key_key
  on credit_grants (billing_cycle_key)
  where billing_cycle_key is not null;
  • Build the cycle key from stable provider truth, not a random request id.
  • Store the Stripe event id separately from the product grant key.
  • Return the existing grant on duplicate delivery instead of minting another balance.

3. Do not acknowledge failed local dispatch as complete

A common webhook bug is inserting a received event row, failing the local grant write, then returning 200 forever because future replays see the event row and no-op. Non-terminal event rows must retry dispatch or move to a repair state.

  • Event row exists with received or failed, but the grant row is missing.
  • Webhook retry returns 200 no-op because only event id existence is checked.
  • Grant fails after payment succeeds and no repair evidence is visible to support.
  • Subscription row says active, but credits/sessions are not available to the user.

4. Add a recency guard before local mirrors overwrite each other

Subscription events can arrive out of order. If an older event writes after a newer event, your local status and grant eligibility can drift away from Stripe.

-- Example shape only.
update subscriptions
set status = :status,
    current_period_end = :current_period_end,
    last_event_at = :stripe_event_created
where stripe_subscription_id = :stripe_subscription_id
  and last_event_at <= :stripe_event_created;
  • Use the provider event creation time or a stronger ordering field when available.
  • Keep the guard atomic in the database write, not only in application code.
  • Log ignored stale events so support can explain why a replay did not change access.

5. Replay the billing path before launch

  • Checkout completed, but no credits granted until the verified webhook path runs.
  • First invoice.paid creates one grant.
  • Duplicate invoice.paid returns the same grant.
  • Grant write fails, webhook retries, and the second attempt completes the grant.
  • Older subscription update arrives after a newer period update and is ignored.
  • Deleted or unpaid subscription stops future grants without deleting historical grant evidence.
