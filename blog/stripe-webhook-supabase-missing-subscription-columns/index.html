Stripe webhook + Supabase schema drift

Your Stripe webhook can fail because the subscription columns are not there.

A common billing regression is not a bad Stripe event. It is local code writing plan limits, included credits, or tier fields into a Supabase table that never had those columns.

If `stripe-webhook` and `check-subscription` disagree with the actual schema, a paid user can hit a failed local write, stale limits, or a silent fallback path that makes Stripe look healthy while the app state is wrong.

The invariant to protect

Every Stripe subscription event should write only to columns that exist on the chosen source-of-truth table, and every derived plan limit should be read from the right config table before access is granted.

Fixed-scope review fit: EUR99 when one Stripe event path and one Supabase subscription source-of-truth row need replay; EUR149 if plan limits, credits, and feature gates are spread across multiple tables.

1. Separate subscription state from plan config

Subscription truth

  • Stripe customer/subscription id
  • Status, period, cancel, and price references
  • Webhook event id and delivery status
  • Processed-event row for duplicate delivery

Plan-limit truth

  • `pricing_config` or plan table row
  • Team member limits and included credits
  • Feature gate resolver output
  • Fallback behavior when config is missing

2. Test the schema-drift failure cases

  • Webhook writes `team_member_limit` or `included_credits` to `subscriptions`, but those columns only exist in `pricing_config`.
  • Manual subscription check updates a different shape than the webhook path.
  • A failed local write is swallowed, leaving Stripe active but Supabase stale.
  • Duplicate webhook delivery retries the same bad write and hides the original event cause.
  • Feature gates read from a denormalized subscription row while plan config changed elsewhere.

3. Replay one event through the exact schema

stripe event received -> processed-event insert succeeds
subscription source row -> only existing columns are updated
plan config lookup -> limits/credits are derived, not invented
feature gate resolver -> reads the same source on webhook and manual check
duplicate event -> returns 2xx without rewriting stale or missing columns
