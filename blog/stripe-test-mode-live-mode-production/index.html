Stripe production launch debug

Stripe works in test mode but fails in live mode?

When checkout works locally or in Stripe test mode but production breaks, the cause is usually not one mysterious Stripe bug. It is usually a mixed set of test/live API keys, price IDs, webhook secrets, portal configuration, Vercel environment variables, or Supabase state updates.

Start by separating test and live artifacts

Stripe test mode and live mode are separate worlds. Do not debug by looking for a generic “Stripe config”. Build a table of every key, price, product, webhook endpoint, customer portal setting, and deployed environment variable that production actually uses.

1. Match every live key with live resources

The most common production failure is mixing live API keys with test resources, or test keys with live price IDs. Stripe object IDs can look similar, so verify them in the dashboard mode where they were created.

  • STRIPE_SECRET_KEY in production should be a live key, not sk_test_....
  • Checkout price_... IDs should exist in live mode and match the product/interval/currency you expect.
  • Publishable key, secret key, webhook secret, and price IDs should all come from the same Stripe account and same mode.
  • If you use multiple products or plans, confirm the code path does not fall back to a test/default price when an env var is missing.

2. Create a live webhook endpoint and use its own signing secret

Local CLI webhook secrets, test-mode dashboard secrets, and live-mode endpoint secrets are different. If production verifies a live webhook with a test or CLI whsec_..., your handler will reject real events before it updates Supabase.

  • Create the production webhook endpoint in Stripe live mode with your deployed URL, for example https://yourdomain.com/api/stripe/webhook.
  • Copy the signing secret from that exact live endpoint into the production environment.
  • Check the endpoint delivery logs in Stripe: HTTP status, response body, redirects, signature errors, and whether events are being sent at all.
  • Make sure the webhook route is public, does not require app auth, and is not redirected by middleware, canonical-domain rules, or trailing-slash redirects.

3. Redeploy after changing Vercel environment variables

Changing env vars in a dashboard does not always change the already-running deployment. Treat env changes like code changes: update production variables, redeploy, then verify the deployed function sees the expected mode.

Production env checklist

  • Live Stripe secret key.
  • Live publishable key exposed to the browser.
  • Live price IDs for each plan.
  • Live webhook signing secret.
  • Canonical app URL used in success/cancel/portal returns.

Drift symptoms

  • Checkout opens but shows the wrong product or amount.
  • Payment succeeds but Supabase still says free/inactive.
  • Portal link fails or opens for the wrong customer.
  • Webhook logs show 307/308 redirects, 401/403, or signature verification failures.

4. Configure the live Customer Portal separately

Stripe Billing Portal settings can differ by mode. A portal flow that works in test mode can fail in live mode if the live portal is not configured, the return URL is wrong, or the customer ID in Supabase points to a test customer.

  • Enable and configure the live Customer Portal in the Stripe dashboard.
  • Confirm your app stores a live cus_... ID after a live checkout, not a test customer ID from earlier testing.
  • Use a live-mode Stripe client when creating the portal session.
  • Check that cancellation, downgrade, and subscription update events are included in webhook handling.

5. Verify Supabase state from webhook events, not success URLs

Success URLs are useful for the user experience, but they are not reliable proof of payment. Production access should unlock from verified Stripe webhook events mapped to the right Supabase user or organization.

  • Store user_id or organization ID in Checkout metadata or map through a trusted customer table.
  • Handle at least checkout.session.completed, customer.subscription.updated, and customer.subscription.deleted for subscriptions.
  • Log event IDs and make updates idempotent so retries do not corrupt state.
  • In Supabase, compare the live customer/subscription IDs against the event payload before blaming frontend gating.

What to send for a production billing review

Do not send secrets. Send sanitized env variable names, Stripe dashboard screenshots with secret values hidden, webhook delivery logs, relevant route files, and the Supabase rows that should change after a live checkout.

Quick diagnosis table

Checkout says price not found

The deployed app is probably using a test price ID with live keys, a live price ID from another account, or an env var that was not redeployed.

Payment succeeds but app stays free

Check live webhook endpoint, signing secret, event delivery, user/customer mapping, and Supabase update logs.

Webhook returns 307 or 308

Use the final canonical webhook URL in Stripe and exclude the webhook route from middleware/canonical redirects.

Portal link fails in production

Check live portal configuration, live customer IDs, return URL, and whether you are creating the portal session with a live-mode Stripe client.
