Stripe portal debug

Stripe Customer Portal cancelled, but Supabase still says active?

Customer Portal bugs are often state-model bugs. The portal can cancel immediately, schedule cancellation at period end, change quantity, switch prices, or fail payment later. Your app needs to sync Stripe subscription events into a small, explicit Supabase access model.

First, name the exact state mismatch

Before changing code, compare what Stripe says now with what your app grants now. A user who set cancel_at_period_end=true may still deserve access until current_period_end. A user with status=canceled probably should not.

1. Decide which Stripe fields drive app access

Do not store only a boolean like is_paid and hope it survives every portal action. Keep enough subscription state to explain the access decision.

  • stripe_customer_id and stripe_subscription_id for reliable lookup.
  • status: active, trialing, past_due, unpaid, canceled, incomplete, etc.
  • cancel_at_period_end, cancel_at, and canceled_at so scheduled cancellations are not treated like immediate cancellations.
  • current_period_end to decide whether access continues after a scheduled cancellation.
  • Current price/product IDs if downgrade, upgrade, or tier gating matters.

2. Handle portal changes through subscription webhooks

Stripe documents subscription webhooks as the place to manage subscription activity because much of it is asynchronous. For portal-driven changes, the app should not depend on a return URL visit to update access.

  • Listen for customer.subscription.updated for scheduled cancels, plan changes, quantity changes, trial changes, and payment-state transitions.
  • Listen for customer.subscription.deleted for immediate cancellation or final cancellation.
  • Keep invoice.payment_failed / invoice.payment_succeeded behavior separate from portal cancel behavior.
  • Make webhook updates idempotent. Events can arrive more than once and not always in the mental order you expected.

3. Treat scheduled cancellation differently from lost access

Usually still has access

  • status=active
  • cancel_at_period_end=true
  • current_period_end is in the future
  • Your product promises access until period end

Usually loses access

  • status=canceled
  • current_period_end has passed
  • Subscription deleted immediately
  • Your product has no grace period for unpaid/canceled state

This is a business rule as much as a code rule. Write it down, then implement it.

4. Reconcile from Stripe when Supabase drifts

If a user reports wrong access, do one manual reconciliation before rewriting the integration.

  • Find the user in Supabase and note customer ID, subscription ID, stored status, period end, cancel flags, and updated timestamp.
  • Open the matching subscription in Stripe and compare status, cancellation fields, current period, current price, and latest invoice state.
  • Inspect webhook delivery logs for the portal action timestamp and confirm your endpoint returned 2xx.
  • If Stripe sent the event and your endpoint returned 2xx but Supabase is stale, debug handler logic/database permissions.
  • If Stripe never sent the expected event to production, debug endpoint mode, selected events, live/test split, or portal test mode.

5. Make the portal return URL informational, not authoritative

The portal return URL is useful UX, but it should not be the source of truth for billing state. After return, show a message like “billing changes may take a few seconds,” then read access from the synced subscription row or fetch Stripe server-side for a narrow reconciliation.

  • Do not unlock or revoke access only because the user visited a success/return URL.
  • Do not trust client-supplied subscription IDs or price IDs for access.
  • Do not collapse active, scheduled-cancel, past_due, unpaid, and canceled into one opaque boolean without keeping raw sync fields.

What to send for a narrow billing review

The fastest useful review includes one affected user, the Stripe subscription ID, sanitized Supabase subscription row, webhook delivery status around the portal action, and your intended access rule.
