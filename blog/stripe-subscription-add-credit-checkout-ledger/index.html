Subscription credits + one-time add-credit purchases

Subscription credits and add-credit checkout can drift.

A common SaaS pattern is simple on the pricing page: Starter, Pro, and Team include monthly credits, and users can buy an extra credit when they need more. The billing-state risk is that two Stripe paths now write to one local balance.

If invoice.paid, checkout.session.completed, a credit grant function, and a monthly reset job disagree, users can lose purchased credits, receive duplicate credits, or see the wrong remaining balance after renewal.

The invariant to protect

Monthly included credits and one-time purchased credits should converge on one idempotent credit ledger. Subscription renewal can reset or refill the included allowance, but it must not overwrite purchased credit or double-apply an add-credit Checkout session.

Evidence packet

For a narrow review, the useful packet is not the pricing-page copy. It is the smallest proof that both Stripe paths write through the same replay-safe local money state.

  • Subscription renewal event: invoice.paid or the subscription Checkout/session event that creates the monthly credit entitlement.
  • One-time add-credit event: checkout.session.completed, PaymentIntent, or invoice/payment object for the add-credit purchase.
  • Credit ledger row: signed delta, balance after, source event id, credit type, and account/customer id.
  • Grant function or webhook log: the code path that mutates the balance for both monthly and purchased credits.
  • Tier config: monthly credits for Starter, Pro, Team, or equivalent plans.
  • Replay proof: duplicate webhook, add-credit retry, renewal boundary, and monthly reset behavior.

1. Model included and purchased credits separately

A single balance is fine if the ledger preserves why each delta exists. Problems start when the subscription refill job rewrites the entire balance without knowing which credits were purchased separately.

Included credits

  • Granted by plan tier
  • Usually period-bound
  • Often reset or refill monthly
  • May expire at renewal boundary

Purchased credits

  • Granted by one-time Checkout
  • Should be additive
  • May survive monthly reset
  • Must be replay-safe by purchase event

2. Use a ledger, not only a balance column

The balance column is a fast read model. The ledger is how you prove what happened after webhook retries, refunds, plan changes, and manual repairs.

-- Example shape only: adapt table names to your app.
create table credit_ledger (
  id bigserial primary key,
  account_id uuid not null,
  source_type text not null,
  source_event_id text not null,
  credit_delta integer not null,
  balance_after integer not null,
  created_at timestamptz not null default now(),
  unique (source_type, source_event_id)
);
  • Use source_type values such as monthly_subscription_credit and add_credit_checkout.
  • Keep source_event_id stable across retries.
  • Store the Stripe event id and the Checkout Session or PaymentIntent id if both matter.
  • Return the existing ledger row on duplicate delivery instead of applying the delta again.

3. Make the reset boundary explicit

Before launch, decide whether a renewal resets only the included allowance, resets the whole visible balance, or adds a monthly grant. Each choice is valid for a different product. Mixing them silently is the bug.

  • Monthly reset sets total balance to 3 and overwrites 2 purchased credits.
  • Add-credit Checkout grants once on success URL and once again on webhook delivery.
  • Plan upgrade changes monthly allowance but does not preserve purchased credits.
  • Refund or dispute removes money but leaves purchased credits active.

4. Replay the exact mixed path

  • Customer subscribes to a plan with 1, 2, or 3 monthly credits.
  • Customer buys one extra credit through one-time Stripe Checkout.
  • Duplicate add-credit webhook returns the existing ledger row.
  • Subscription renews after purchased credits remain unused.
  • Monthly reset updates included credits without deleting purchased credits.
  • Payment succeeds but local ledger write fails, then webhook retry completes the grant.
  • Refunded add-credit purchase creates one signed reversal or repair state.

What a focused review returns

A first pass should return the exact drift point, the replay matrix, and the smallest ledger/reset change needed to keep monthly and purchased credits consistent. It should not require production secrets or a broad rebuild.
