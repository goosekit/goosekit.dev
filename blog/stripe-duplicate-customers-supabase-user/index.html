Stripe customer mapping + Supabase access

Stripe duplicate customers breaking Supabase access?

A user can complete checkout, pay successfully, and still look unpaid in your app when the Stripe customer or subscription is not mapped to the same Supabase user or organization every time.

The production failure: “Stripe shows payment, Supabase cannot find the user.”

Duplicate Stripe customers usually appear after retries, multiple checkout attempts, payment links, email-only matching, or a database rollback after creating a Stripe customer. The result is messy: one customer has the payment, another customer is stored on the app user, and webhooks update the wrong row or no row at all.

Need customer-mapping help? User mapping checklist

Founding setup/debug help starts at €99–149 when the failing Checkout Session, Stripe customer IDs, and Supabase mapping schema are narrow enough to review safely.

1. Reuse the customer ID before creating Checkout

For logged-in SaaS checkout, the safest default is: look up the Supabase user or organization, reuse its stored Stripe customer ID, and pass that ID into Checkout. Do not rely on the email typed into Checkout to magically merge customers later.

  • Store stripe_customer_id on the billing owner: user, team, workspace, or organization.
  • When creating a subscription Checkout Session, pass customer: stripeCustomerId if the owner already has one.
  • Only create a new Stripe customer when the owner genuinely has no Stripe customer ID yet.
  • Persist the new Stripe customer ID immediately, before creating unrelated side effects that could fail.
  • Use client_reference_id and subscription/customer metadata as backup context, not as the only mapping.

2. Add idempotency around customer creation

Duplicate customers often come from retries or two browser tabs racing through checkout at the same time. External Stripe calls and database writes need a stable key and a unique local constraint.

// Shape, not drop-in code
const ownerId = workspace.id;
const existing = await getBillingOwner(ownerId);

const customerId = existing.stripe_customer_id
  ?? await createStripeCustomerAndPersist({
    ownerId,
    email: user.email,
    idempotencyKey: `stripe-customer:${ownerId}`
  });

const session = await stripe.checkout.sessions.create({
  mode: 'subscription',
  customer: customerId,
  client_reference_id: ownerId,
  subscription_data: {
    metadata: { owner_id: ownerId, supabase_user_id: user.id }
  },
  line_items: [{ price: priceId, quantity: 1 }],
  success_url,
  cancel_url
});
  • Make stripe_customer_id unique in Supabase so two owners cannot claim the same Stripe customer.
  • Use a stable idempotency key for customer creation, derived from the billing owner, not a random request ID.
  • Lock or upsert the billing-owner row before parallel checkouts can create two customers.
  • Prefer returning the existing active checkout/session path over creating a new customer on every click.

3. Do not repair identity from email alone

Email is useful for support, but it is not a durable billing foreign key. Users change email, teams share billing emails, test/live modes differ, and Stripe can legitimately contain multiple customers with the same email.

Good primary keys

stripe_customer_id, stripe_subscription_id, Supabase user_id, organization/workspace ID, and metadata written when Checkout is created.

Weak fallback only

Email, customer name, Checkout success URL visits, or “latest customer with this email” searches. Use these for manual diagnosis, not automatic access grants.

4. Store enough fields to untangle duplicate customers

When duplicates already exist, you need to compare Stripe customer IDs, subscription IDs, Checkout Session IDs, and the Supabase owner row without exposing secrets.

alter table public.billing_customers add column if not exists owner_id uuid;
alter table public.billing_customers add column if not exists stripe_customer_id text unique;
alter table public.subscriptions add column if not exists owner_id uuid;
alter table public.subscriptions add column if not exists stripe_customer_id text;
alter table public.subscriptions add column if not exists stripe_subscription_id text unique;
alter table public.subscriptions add column if not exists checkout_session_id text;
alter table public.subscriptions add column if not exists last_stripe_event_id text;
alter table public.subscriptions add column if not exists mapping_source text;
  • Record whether a mapping came from an existing stored customer, Checkout metadata, subscription metadata, or manual repair.
  • Keep live-mode and test-mode customer IDs separate; do not let test data overwrite production billing state.
  • Use a server-only Supabase client for webhook writes so RLS does not block billing repair updates.
  • Keep an event ledger so a retry cannot create a second subscription row for the same Stripe subscription.

5. Test the duplicate-customer paths

Most teams test one happy checkout. The duplicate bugs appear when customers retry, open multiple tabs, or switch from test flows to production billing.

  • Click checkout twice quickly and confirm only one Stripe customer is attached to the billing owner.
  • Cancel Checkout, retry, then complete payment and confirm the same customer ID is reused.
  • Start checkout from two devices or tabs with the same logged-in user and confirm unique constraints hold.
  • Complete checkout with the same email on a different Supabase account and confirm your app does not grant access by email alone.
  • Simulate webhook retry and confirm the same subscription row is updated, not duplicated.

What to send for a duplicate-customer review

Do not send secrets. Send sanitized IDs and screenshots that show the mismatch between Stripe and Supabase.
