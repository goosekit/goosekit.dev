Stripe trials + Supabase access

Stripe trial ended but Supabase access did not update?

Trial transitions are a common place for SaaS billing state to drift. Your app may keep trial users active after payment fails, or lock out a customer whose first invoice was paid but whose Supabase row never changed.

The production failure: “trialing” is not your product policy

A Next.js + Supabase app often starts with one flag like is_paid. That breaks down when Stripe moves a subscription from trialing to active, incomplete, past_due, or canceled. Treat trial end as a billing lifecycle, not a checkout-success event.

1. Store trial status separately from access

Do not collapse trial state, paid state, and feature access into one boolean. Store what Stripe says, then map it to what your product allows.

Stripe snapshot

status, trial_end, current_period_end, latest invoice, payment intent state, cancellation state, and customer ID.

Product access

Trial access, full access, grace/limited access, locked access, and the message users see when payment is required.

  • Keep stripe_status and access_state as different fields.
  • Record trial_ends_at and trial_converted_at so you can explain what happened later.
  • Store the Stripe customer/subscription IDs; never rely on email as the billing foreign key.
  • Make the trial-end user message explicit: “add payment method”, “payment failed”, or “subscription active”.

2. Handle the events around trial end

Stripe events are asynchronous. The trial ending warning, subscription update, invoice creation, invoice payment, and payment failure may not arrive in the simple order your UI assumes.

  • customer.subscription.trial_will_end: send or schedule your own reminder if your product needs one.
  • customer.subscription.updated: sync trial_end, status, current price, cancellation, and current period dates.
  • invoice.paid: mark conversion only after the first paid invoice is actually paid, not merely because the trial date passed.
  • invoice.payment_failed: start grace/limited access, billing warnings, or lockout depending on your policy.
  • customer.subscription.deleted: remove or downgrade access when the trial/subscription is actually canceled.

3. Use a policy table for trial transitions

The exact policy depends on your product and Stripe trial settings. The key is to make it testable instead of scattered across route handlers.

const accessPolicy = {
  trialing: { access: 'trial', showTrialEndsAt: true },
  active: { access: 'full' },
  incomplete: { access: 'limited', showPaymentRequired: true },
  past_due: { access: 'grace', showBillingBanner: true },
  unpaid: { access: 'limited', showBillingBanner: true },
  canceled: { access: 'locked' },
  incomplete_expired: { access: 'locked' }
};

The names are examples. What matters is that trial ending, first invoice payment, and failed payment each have a deliberate product-access result.

4. Test the edge cases before you trust the funnel

  • Trial ends and first invoice is paid: access should become full without manual database edits.
  • Trial ends and first invoice fails: access should follow your grace/limited/locked policy.
  • User adds a payment method late: recovery should restore access from a later paid invoice or refreshed subscription snapshot.
  • User cancels during trial: access should not remain full just because trial_end is in the future.
  • Older webhook retries must not overwrite newer conversion or cancellation state.

What to send for a trial/access review

Send artifacts, not secrets: event types, relevant timestamps, subscription status, invoice status, and the Supabase table shape that stores access.
