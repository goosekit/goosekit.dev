Stripe webhook delivery status debug

Stripe webhook gets 307, 308, 401, or 403 in Vercel production?

If Stripe says your webhook endpoint failed in production, do not start by rewriting subscription logic. First prove that Stripe can POST to the exact endpoint URL without being redirected, authenticated like a browser user, blocked by middleware, or parsed before signature verification.

A webhook endpoint is not a protected app route

Stripe needs a public POST URL that returns a fast 2xx after verifying the stripe-signature header. Redirects and login walls that are harmless for browser pages can break webhook delivery completely.

Need a production webhook pass? Paid-state checklist

Fixed-scope setup help starts at €199 when the endpoint, delivery logs, and route files are narrow enough to inspect safely.

1. Match the Stripe endpoint URL to the deployed route exactly

In App Router, the production route is usually a route.ts file such as app/api/stripe/webhook/route.ts, which maps to /api/stripe/webhook. Stripe must call that exact production URL.

  • Check the endpoint in Stripe Dashboard, not only your code comments.
  • Use the canonical production host: avoid old Vercel branch domains, localhost, or non-www/www mismatch.
  • Remove trailing slash ambiguity. If Vercel or Next.js redirects /api/stripe/webhook/ to /api/stripe/webhook, Stripe may record a 307/308 instead of a success.
  • Confirm the route accepts POST. A GET health check returning 200 does not prove Stripe POST delivery works.

2. Treat 307/308 as URL canonicalization bugs first

A 307 or 308 usually means the request reached something that wants a different URL: http to https, apex to www, www to apex, slash to no-slash, or an app-level redirect. Stripe webhook endpoints should not depend on following redirects.

Common redirect causes

  • Endpoint uses example.com while Vercel redirects to www.example.com, or the reverse.
  • Trailing slash setting changes the route URL.
  • Middleware redirects unknown unauthenticated paths to login.
  • A root-domain redirect rule catches /api/* accidentally.

Fast check

curl -i -X POST https://example.com/api/stripe/webhook

# You want no Location header redirect.
# A 400 for missing Stripe signature is okay.
# A 307/308 means fix the URL/routing first.

3. Treat 401/403 as middleware or auth guard leaks

Your webhook handler should verify Stripe's signature, not require a Supabase session, Clerk session, NextAuth user, CSRF cookie, or admin role. If middleware protects all routes by default, explicitly exclude the webhook path.

// middleware.ts / proxy.ts matcher example
export const config = {
  matcher: [
    '/((?!api/stripe/webhook|_next/static|_next/image|favicon.ico).*)',
  ],
};
  • If Stripe delivery says 401, search middleware, route wrappers, auth helpers, and API protection utilities.
  • If Stripe delivery says 403, check role gates, CSRF checks, bot protection, firewall rules, and platform protection settings.
  • Do not solve this by disabling signature verification. The endpoint is public, but only signed Stripe events should be trusted.

4. Verify raw body and runtime after the route is reachable

Once the status is no longer redirect/unauthorized, then debug signature errors. Stripe signature verification needs the unmodified request body and the correct webhook signing secret for that exact endpoint and mode.

export async function POST(req: Request) {
  const body = await req.text();
  const signature = req.headers.get('stripe-signature');

  if (!signature) {
    return new Response('Missing stripe-signature', { status: 400 });
  }

  const event = stripe.webhooks.constructEvent(
    body,
    signature,
    process.env.STRIPE_WEBHOOK_SECRET!
  );

  // handle event idempotently...
  return Response.json({ received: true });
}
  • Use the Dashboard endpoint signing secret in production; the Stripe CLI secret is only for local forwarding.
  • Keep test-mode and live-mode endpoint secrets separate.
  • Read req.text() once before parsing JSON.
  • Return quickly; move slow email/analytics work out of the critical webhook response path.

5. Use Stripe delivery logs as the source of truth

The browser success page can load while the webhook is still failing. For billing state, inspect Stripe's event delivery attempts and your Vercel function logs for the same timestamp.

In Stripe

  • Endpoint URL and mode: test or live.
  • HTTP status: 307, 308, 401, 403, 400, 500, or timeout.
  • Event type and event ID.
  • Response body or redirect Location if shown.

In Vercel / app logs

  • Did the function execute at all?
  • Was the route handler reached before middleware?
  • Was stripe-signature present?
  • Was the error signature verification, database write, or app authorization?

What to send for a webhook delivery review

Do not send secrets. Send the Stripe endpoint URL, recent delivery status screenshot, route path/files, middleware matcher, Vercel redirect/domain settings summary, and sanitized webhook handler.

Quick diagnosis table

307 or 308

Fix the endpoint URL and redirects first: canonical domain, trailing slash, HTTPS, and any middleware/app redirect touching /api/*.

401

Your webhook path is probably protected by app auth, an API helper, or middleware. Exclude the path and keep Stripe signature verification inside the handler.

403

Look for role gates, CSRF checks, bot/firewall protection, deployment protection, or a middleware branch returning forbidden.

400 signature error

Now debug raw body and signing secret: correct endpoint secret, test/live mode, and no JSON parsing before constructEvent.
