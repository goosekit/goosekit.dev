Stripe Customer Portal + Supabase entitlements

Customer Portal plan changes leaving Supabase on the wrong plan?

When a customer upgrades, downgrades, schedules a change, or cancels at period end in Stripe Customer Portal, your local Supabase row needs more than a simple status = active check. The hard part is deciding what access is active now, what is pending, and what should happen at the next period boundary.

The production failure: “Stripe is correct, my app entitlement is not”

Portal changes are not just checkout again. A downgrade can be immediate or scheduled. A cancellation can set cancel_at_period_end while the customer should still keep access until current_period_end. A schedule can represent the next plan before it is active. If Supabase stores only one plan and one is_paid boolean, the UI eventually lies.

1. Classify the Portal change before updating entitlements

Start by separating the Stripe billing state from your product access state. These are related, but they are not identical.

Immediate upgrade

The subscription item or price changes now. Your app can usually grant the new higher entitlement as soon as Stripe reports the updated subscription and payment state is safe.

Scheduled downgrade

The current plan remains active until the period ends, while a lower plan is pending through a schedule, pending update, or next-phase change.

Cancel at period end

cancel_at_period_end = true does not always mean “remove access now”. It usually means keep current access until current_period_end.

Immediate cancellation or failed payment

Now the entitlement decision depends on status, collection state, grace period policy, and whether you intentionally keep limited access.

  • Log sanitized subscription.id, status, price.id, current_period_end, cancel_at_period_end, cancel_at, canceled_at, and schedule.
  • Record which Portal configuration is enabled: immediate updates, prorations, scheduled changes, cancellations, or payment-method-only changes.
  • Do not infer product access from a Portal return URL. Treat the return URL as navigation, not proof of a completed billing change.
  • Use webhook events and a Stripe Subscription fetch as the source for final local state.

2. Store current entitlement and pending entitlement separately

A single plan column cannot represent “Pro until July 1, then Starter” without ambiguity. Keep the active access window and pending change explicit.

alter table public.subscriptions add column if not exists current_price_id text;
alter table public.subscriptions add column if not exists current_plan_key text;
alter table public.subscriptions add column if not exists entitlement_active_until timestamptz;
alter table public.subscriptions add column if not exists cancel_at_period_end boolean default false;
alter table public.subscriptions add column if not exists pending_price_id text;
alter table public.subscriptions add column if not exists pending_plan_key text;
alter table public.subscriptions add column if not exists pending_effective_at timestamptz;
alter table public.subscriptions add column if not exists last_stripe_event_id text;
alter table public.subscriptions add column if not exists last_stripe_event_created_at timestamptz;
  • current_plan_key answers: what should the user have access to right now?
  • entitlement_active_until answers: until when is this access valid if nothing else changes?
  • pending_plan_key answers: what should happen at the period boundary?
  • cancel_at_period_end answers: should the UI show “access ends on this date” instead of “subscription cancelled now”?

3. Let subscription events own Portal changes

For most SaaS apps, the Portal flow should redirect the customer back to your app, then your app should wait for customer.subscription.updated, customer.subscription.deleted, invoice.paid, or a safe Subscription fetch before changing durable entitlements.

async function applySubscriptionSnapshot(subscription, event) {
  const currentItem = subscription.items.data[0];
  const periodEnd = new Date(subscription.current_period_end * 1000).toISOString();

  const patch = {
    stripe_subscription_id: subscription.id,
    stripe_customer_id: subscription.customer,
    stripe_status: subscription.status,
    current_price_id: currentItem.price.id,
    current_plan_key: planKeyForPrice(currentItem.price.id),
    entitlement_active_until: periodEnd,
    cancel_at_period_end: subscription.cancel_at_period_end,
    last_stripe_event_id: event.id,
    last_stripe_event_created_at: new Date(event.created * 1000).toISOString()
  };

  // If your Portal/schedule flow exposes a future phase, store it as pending
  // rather than overwriting current_plan_key before it is actually active.
  await upsertSubscriptionMonotonically(patch);
}

The exact event set depends on your Portal configuration. The invariant is simpler: local entitlements should be derived from a current Stripe subscription snapshot plus your own access policy, not from the fact that a user clicked “Return”.

4. Make the UI honest about period-end changes

The customer-facing bug is often a trust bug: the billing portal says one thing and the app dashboard says another.

Cancellation pending

Show “Your Pro access ends on June 30” instead of “Free plan” while cancel_at_period_end is true and current_period_end is in the future.

Downgrade pending

Show “Pro now, Starter from June 30” instead of overwriting the current plan immediately.

Upgrade completed

Show the new higher plan once the subscription snapshot and payment state support granting access.

Payment action needed

Show a billing issue state when Stripe status requires payment action, instead of silently leaving old access forever.

5. Check the four common Supabase mistakes

  • Do not set is_paid = false immediately just because cancel_at_period_end is true.
  • Do not overwrite an active plan with a future downgrade before current_period_end or the schedule phase transition.
  • Do not key subscription rows by email. Use Stripe customer/subscription IDs and trusted internal user/org mapping.
  • Do not process Portal updates with a Supabase anon/session client if RLS blocks billing-table writes.
  • Do not let older webhook retries overwrite newer subscription state; keep event IDs and Stripe event timestamps.

What to send for a Portal entitlement review

Do not send secrets. Send the facts needed to reproduce the mismatch between Stripe and Supabase.
