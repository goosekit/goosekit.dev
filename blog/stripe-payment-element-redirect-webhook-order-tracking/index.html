Stripe Payment Element + async fulfillment

Stripe Payment Element redirect but order tracking drifted?

Payment Element often returns the customer before the webhook has finished the real work. That is fine if the order tracker treats the redirect as a hint and the webhook as the source of truth.

The failure mode is a diner or buyer who sees redirect_status=succeeded while the live tracking page is still missing the order, showing stale status, or trusting the client-side PaymentIntent result too early.

The invariant to protect

The return page can read payment_intent and redirect_status, but order fulfillment, paid status, and Realtime tracking should converge from webhook-owned state.

Fixed-scope review fit: EUR149 when Payment Element redirect, webhook fulfillment, order rows, and live tracking can disagree for paid orders.

1. Separate redirect state from fulfillment state

Redirect hint

  • return_url receives payment_intent and redirect_status
  • Client page can show pending confirmation or tracking shell
  • Client must not mark the order paid by itself
  • Refresh/reload should not create another order mutation

Webhook truth

  • payment_intent.succeeded or Checkout event updates the order row
  • Processed-event row proves duplicate delivery is safe
  • Order status transition is idempotent and replayable
  • Realtime/subscription payload matches the stored order state

2. Test the states that buyers actually see

  • Redirect succeeds before webhook delivery: tracking page should show pending, then update from the order row.
  • Webhook succeeds before redirect loads: tracking page should render paid/confirmed immediately from stored state.
  • Webhook retries: the order row and tracking event should not duplicate fulfillment or notifications.
  • PaymentIntent fails or is canceled after a redirect attempt: tracking should fail closed, not show confirmed.
  • Realtime disconnects: refresh should recover from the database row, not from stale client params.

3. Use one replay matrix for redirect, webhook, and tracking

The review should prove what each layer is allowed to write. Keep the matrix small: one paid order, one failed order, one duplicate webhook, and one tracking refresh.

redirect_status=succeeded + webhook pending -> tracking shows pending from order row
webhook succeeded + redirect not loaded -> tracking shows paid/confirmed from order row
duplicate payment_intent.succeeded -> processed-event hit, no second fulfillment
redirect success + payment failed later -> order remains unpaid/failed
Realtime missed event + page refresh -> state recovers from database truth
