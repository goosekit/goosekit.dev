Supabase auth callback works locally but fails on Vercel?

This failure pattern is usually not mysterious: localhost and production disagree on the callback URL, cookie behavior, middleware, environment variables, or OAuth provider settings. Use this checklist before rewriting your auth flow.

Start with the environment split

If login works on localhost:3000 but fails after deploying, compare the exact URL chain in production: app URL → Supabase auth URL → provider callback → your /auth/callback route → final dashboard redirect.

1. Check Supabase Site URL and redirect allowlist

  • Set Supabase Site URL to the canonical production app URL, not localhost or an old Vercel preview domain.
  • Add explicit redirect URLs for local and production, for example http://localhost:3000/auth/callback and https://yourdomain.com/auth/callback.
  • If you use Vercel preview deployments, add a safe wildcard only for previews you actually need. Keep production explicit.
  • Confirm the path casing and trailing slash match what your app uses.

2. Stop hardcoding localhost in auth actions

Many production bugs survive dashboard checks because the app still sends a local redirect in code.

  • Search for localhost, NEXT_PUBLIC_SITE_URL, VERCEL_URL, and redirectTo in login, signup, magic-link, and OAuth calls.
  • Build absolute redirects from a single trusted site URL helper per environment.
  • Do not use window.location.origin inside server actions or server components.
  • In Vercel, remember VERCEL_URL usually has no protocol. Normalize it before using it in redirect URLs.

3. Verify the callback route actually exchanges the code

For App Router projects using Supabase SSR helpers, the callback route needs to read the code, exchange it for a session, set cookies, then redirect.

Capture from production

  • Full callback URL after auth
  • Presence of code= or error=
  • Vercel function logs for callback route
  • Final redirect destination

Common route issues

  • Callback route exists locally but is missing in production
  • Code exchange not awaited
  • Cookies set after a redirect response is already created
  • Edge/runtime mismatch with the auth helper used

4. Separate OAuth provider config from Supabase config

  • For Google/GitHub/etc., confirm the provider callback points to the Supabase auth callback endpoint, not directly to your app callback route unless your flow expects that.
  • Check both test and production OAuth apps. It is easy to update one and deploy with credentials from the other.
  • Confirm client IDs/secrets in Vercel match the provider app whose callback URLs you edited.
  • If only one provider fails, debug provider config first. If magic links and OAuth both fail, debug Supabase redirect/app cookies first.

5. Test cookies, middleware, and protected routes after login

Sometimes the callback succeeds but middleware immediately treats the user as logged out.

  • Inspect production cookies after callback. Confirm auth cookies are set for the right domain and path.
  • Check middleware uses the same Supabase SSR pattern as the rest of the app and refreshes sessions correctly.
  • Test in a clean browser profile so stale local cookies do not hide a production bug.
  • Log the authenticated user ID server-side on the first protected route, then remove noisy logs after fixing.

What to send for a narrow review

The fastest useful review includes the production callback URL, Supabase redirect settings, relevant auth files, Vercel env names, provider callback settings, and one failing login attempt timestamp.
