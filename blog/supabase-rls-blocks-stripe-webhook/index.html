Supabase RLS + Stripe webhook debug

Supabase RLS blocking your Stripe webhook writes?

If Stripe delivery is 200-ish or your handler logs show the event was verified, but Supabase still refuses to insert/update subscriptions, profiles, or organizations, you may be writing from the wrong Supabase client. A webhook has no logged-in browser user, so user-scoped RLS policies often block it.

The common failure: valid Stripe event, wrong Supabase authority

Stripe webhooks are server-to-server. The request is not signed in as the app user who started Checkout. If the handler uses an anon key, publishable key, or cookie/session client, Supabase may apply RLS as anon or the wrong user and reject the database write.

Need webhook paid-state help? User mapping checklist

Founding setup/debug help starts at €99–149 when the failing route, table schema/policies, logs, and desired paid-state update are narrow enough to inspect safely.

1. Separate Stripe success from Supabase write success

A successful Checkout session or a verified webhook signature only proves Stripe sent a valid event. It does not prove your database update ran with the right Supabase role, mapped to the right user, or passed table constraints.

  • Check Stripe delivery logs: did the event reach the endpoint and return 2xx?
  • Check Vercel/function logs around the Supabase insert, upsert, or update.
  • Log sanitized Supabase errors: code, message, details, hint. Do not swallow error.
  • Confirm the failing table has RLS enabled and policies that would block anon/authenticated writes.

2. Use a server-only Supabase client for webhook writes

Supabase's public docs emphasize that RLS protects exposed schemas and that elevated keys belong only in backend components. For Stripe webhook writes, the usual pattern is a dedicated server-side client using a service role / secret key kept out of the browser bundle.

Webhook server client pattern

import { createClient } from '@supabase/supabase-js';

const supabaseAdmin = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,
  { auth: { persistSession: false } }
);

// After Stripe signature verification:
await supabaseAdmin
  .from('subscriptions')
  .upsert({ user_id, stripe_customer_id, status });

Avoid in webhooks

  • A browser/client Supabase instance using the anon or publishable key.
  • A server component/helper that reads cookies and expects a logged-in user.
  • Passing service role keys to client components, NEXT_PUBLIC_ variables, or shipped JavaScript.
  • Creating broad RLS policies just to make the webhook pass.

3. Keep the service role powerful but narrow in code

A service role can bypass RLS, which is useful for trusted backend work and dangerous if treated casually. The safer fix is not “disable RLS”; it is “use elevated authority only inside the verified webhook path, then write the smallest possible state change”.

  • Verify the Stripe signature before any database mutation.
  • Resolve the internal user_id or organization_id from trusted metadata/customer mapping, not from arbitrary request input.
  • Upsert only the billing fields needed for access: Stripe IDs, status, current period, price/plan, and timestamps.
  • Make webhook processing idempotent so retries do not duplicate subscription rows.

4. Diagnose by error shape

“new row violates row-level security policy”

The handler is probably using anon/authenticated authority or a policy that does not allow that role/action. For a trusted Stripe webhook, use a server-only elevated client after signature verification.

“permission denied for table”

Check database grants and which role the API key maps to. RLS policy fixes will not help if the role lacks table privileges.

No error, but no row changed

Inspect returned data, error, filters, maybeSingle(), and whether your update matched any rows.

Works locally, fails in production

Check that SUPABASE_SERVICE_ROLE_KEY exists in the production environment, was not prefixed with NEXT_PUBLIC_, and the app was redeployed after changes.

5. Do not “fix” this by opening client-side policies

It is tempting to create a policy that lets authenticated users update their own paid status. That can create a privilege escalation path if the browser can write fields that should only be controlled by Stripe. Keep customer-facing reads separate from webhook-owned writes.

  • Browser clients can read their own access/subscription state through strict SELECT policies.
  • Stripe-owned fields should be written by the verified backend webhook path.
  • Admin/support operations should use separate backend routes with explicit authorization, not client-side service keys.

What to send for a Supabase RLS + Stripe webhook review

Do not send secrets. Send the evidence that proves whether the failure is role, policy, env, mapping, or constraint related.
