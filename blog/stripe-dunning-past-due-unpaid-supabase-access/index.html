Stripe dunning + Supabase access

Stripe past_due or unpaid leaving Supabase access wrong?

Failed renewals are not a single boolean. A subscription can be active, past_due, unpaid, canceled, or recovering through Smart Retries while your SaaS needs a separate product-access decision in Supabase.

The production failure: “Do I revoke access now or wait?”

Many Next.js + Supabase apps start with is_paid = subscription.status === 'active'. That is too blunt for dunning. It can lock out real customers during a recoverable card failure, or keep non-paying customers active forever after Stripe moves the invoice to unpaid or the subscription is canceled.

1. Separate Stripe billing status from product access

Stripe status tells you what is happening in billing. Your product access policy decides what the customer can use right now. Store both instead of collapsing them into one is_paid flag.

Billing state

Subscription status, invoice status, retry schedule, latest payment failure, collection method, cancellation state, and whether payment recovery is still possible.

Access state

Current plan, feature limits, grace period end, readonly/limited mode, hard lockout date, and the user-facing billing warning.

  • Record stripe_status exactly as Stripe sends it; do not overwrite it with your own labels.
  • Record access_state separately: full, grace, limited, locked, or whatever names match your product.
  • Record grace_until and locked_at explicitly so retries and late webhooks do not create ambiguous access.
  • Show billing warnings from local state, but let Stripe-hosted invoice/payment-method flows handle payment collection.

2. Treat dunning as a policy table, not scattered if statements

The right answer depends on your product. A B2B app may keep access for 7 days. A compute-heavy API may limit usage immediately. A paid community might keep read-only access. Make that policy explicit.

const accessPolicy = {
  trialing: { access: 'full' },
  active: { access: 'full' },
  past_due: { access: 'grace', graceDays: 7, showBillingBanner: true },
  unpaid: { access: 'limited', showBillingBanner: true },
  paused: { access: 'limited', showBillingBanner: true },
  canceled: { access: 'locked' },
  incomplete: { access: 'locked' },
  incomplete_expired: { access: 'locked' }
};

Do not copy this policy blindly. It is a safe shape, not universal business advice. The important part is that billing status, grace periods, and feature access are intentionally mapped and tested.

3. Listen to the events that actually move dunning

For subscriptions, Stripe recommends handling webhook events because billing changes are asynchronous. Dunning usually involves invoice and subscription events, not just Checkout success.

  • invoice.payment_failed: start or refresh the billing-warning/grace path; store the invoice ID and failure context you can safely show.
  • invoice.paid: clear dunning warnings, restore full access if the subscription snapshot supports it, and update period dates.
  • customer.subscription.updated: sync status, current_period_end, cancel_at_period_end, pause_collection, and current price.
  • customer.subscription.deleted: lock or downgrade access according to your cancellation policy.
  • Fetch the current Stripe Subscription before final writes when event ordering or retry timing could make a stale event misleading.

4. Store enough Supabase fields to recover cleanly

Dunning bugs become much easier to debug when the row explains both the current billing snapshot and the product-access decision.

alter table public.subscriptions add column if not exists stripe_status text;
alter table public.subscriptions add column if not exists access_state text default 'locked';
alter table public.subscriptions add column if not exists grace_until timestamptz;
alter table public.subscriptions add column if not exists current_period_end timestamptz;
alter table public.subscriptions add column if not exists latest_invoice_id text;
alter table public.subscriptions add column if not exists latest_payment_failed_at timestamptz;
alter table public.subscriptions add column if not exists payment_recovered_at timestamptz;
alter table public.subscriptions add column if not exists last_stripe_event_id text;
alter table public.subscriptions add column if not exists last_stripe_event_created_at timestamptz;
  • Use a server-only Supabase client for webhook writes; RLS and user-scoped clients often block billing-table updates.
  • Use Stripe customer/subscription IDs for mapping; email is not a reliable billing foreign key.
  • Keep idempotency/event ledger logic so Stripe retries do not duplicate or roll back access state.
  • Keep older events from overwriting newer subscription snapshots by comparing event timestamps or fetching fresh from Stripe.

5. Test the unhappy paths before launch

Checkout success is the easiest case. The revenue leaks and support tickets usually appear when payment fails after the first month.

  • Test a failed renewal and confirm the app shows a billing warning instead of silently pretending everything is fine.
  • Test successful recovery after a failed invoice and confirm access returns without manual database edits.
  • Test what happens after all retries fail: canceled, unpaid, or past_due depending on your Stripe Billing setting.
  • Test an old invoice.payment_failed retry after a newer invoice.paid event; it must not relock access.
  • Test user-facing copy: “payment issue, update your card” is clearer than “your account is unpaid”.

What to send for a dunning/access review

Do not send secrets. Send the minimum artifacts needed to compare Stripe billing state with Supabase product access.
