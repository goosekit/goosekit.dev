Next.js + Supabase middleware redirect loop after login?

If a user logs in successfully but protected pages still bounce back to /login, the problem is usually cookie propagation, middleware matching, callback handling, or redirect logic. Debug it as a request/response loop, not as a generic “Supabase is broken” problem.

First, capture the loop path

Open the Network tab and record the exact sequence: login action → Supabase callback → your callback route → protected route → middleware redirect. A loop only makes sense when you can see which request has the session cookie and which request loses it.

Callback URL checklist

Founding setup/debug help starts at free when the repo, callback route, middleware file, and failure path are narrow enough to review safely.

1. Exclude auth and callback routes from protection

The fastest way to create a loop is protecting the route that is supposed to finish login.

  • Do not redirect unauthenticated users away from /login, /signup, /auth/callback, password recovery, or public marketing pages.
  • Check your middleware matcher. It should skip static assets and should not blindly run protected-route logic on every path.
  • If you redirect logged-in users away from login, make sure that logic does not also run while the callback is still writing cookies.
  • Keep the callback route boring: exchange the code, set cookies through the SSR client, then redirect once to the intended destination.

2. Verify the cookie is written before the protected route check

With Supabase SSR, server-side auth depends on request cookies. If the callback exchanges the code but the next protected request does not carry the sb-...-auth-token cookie, middleware will think the user is still anonymous.

  • Use the current @supabase/ssr server client pattern for route handlers and middleware/proxy.
  • Confirm the auth cookie appears on the response from your callback route, then appears on the following request to the protected page.
  • Do not mix old auth-helper clients, browser-only clients, and SSR clients in the same callback/middleware flow without a reason.
  • On Vercel/production, check domain, protocol, and SameSite behavior. A callback to one hostname cannot reliably set auth for a different canonical hostname.

3. Use a trustworthy server-side session check

Supabase documentation warns against trusting raw cookie session state in server code. In newer SSR patterns, the middleware/proxy refreshes and validates auth state so server components receive consistent cookies.

Good signs

  • Middleware creates a per-request server client.
  • It refreshes/validates auth and passes updated cookies to both request and response.
  • Protected data uses a server-verified user/claims check, not a client-side flag.
  • Redirect decisions are path-aware and one-directional.

Loop-prone signs

  • Middleware reads a stale cookie and redirects before refresh.
  • Callback uses one Supabase client pattern while middleware expects another.
  • Login redirects to dashboard before cookies are actually set.
  • Every request, including login/callback, is forced through protected-route logic.

4. Keep the “next” redirect safe and finite

Returning users to the page they wanted is useful, but a bad next parameter can preserve the loop forever.

  • Only allow same-origin relative paths such as /dashboard. Never trust arbitrary absolute redirect URLs.
  • If the originally requested path was /login, /auth/callback, or an error route, replace it with a sane default.
  • Clear one-time auth query params after the callback succeeds.
  • Log the resolved destination during debugging so you can see whether middleware and callback disagree.

What to send for a quick review

Do not send secrets. Send the middleware/proxy file, callback route, login action, Supabase client helpers, matcher config, and a sanitized Network screenshot showing the redirect chain and cookie headers.

Quick diagnosis table

Loop only in production

Check Site URL, redirect allowlist, callback hostname, cookie domain, HTTPS, and Vercel preview vs production env variables.

Loop after OAuth only

Check provider callback URL, Supabase redirect URL, and whether the callback route exchanges the code before middleware evaluates the dashboard.

Loop after magic link only

Check email link redirect target, token/code handling, and whether the link opens a different hostname than the app expects.

Loop after refresh/token expiry

Check middleware/proxy refresh behavior, request/response cookie updates, and whether server actions depend on middleware redirects.
