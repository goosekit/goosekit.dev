Stripe webhook signature debug

Stripe webhook signature verification fails only in production?

If the same handler works locally with stripe listen but fails on Vercel with “No signatures found matching the expected signature for payload”, debug the three inputs before touching billing logic: the exact raw request body, the Stripe-Signature header, and the endpoint signing secret used by that deployed endpoint.

Most production signature failures are configuration drift

Stripe signs the raw payload for a specific endpoint. A Dashboard webhook secret, Stripe CLI forwarding secret, test-mode endpoint, and live-mode endpoint can all start with whsec_ and still be different secrets.

Need a production webhook pass? Webhook status checklist

Founding setup/debug help starts at €99–149 when the endpoint, logs, route file, and env-target evidence are narrow enough to inspect safely.

1. Confirm you are using the production Dashboard endpoint secret

Stripe's own signature troubleshooting says the common error is using the wrong endpoint secret. The secret printed by stripe listen is for local forwarding. It should not verify events sent by a Dashboard-managed production endpoint.

  • Open the exact webhook endpoint in Stripe Dashboard and reveal the signing secret there.
  • Confirm the endpoint mode: test-mode and live-mode webhook endpoints have separate secrets.
  • Confirm the URL: if you have multiple endpoints, the secret must belong to the endpoint that is receiving the failing event.
  • Check Vercel Production environment variables, not Preview or Development, then redeploy if the value changed.

2. Prove the handler reads the unmodified raw body

The signature is calculated over the exact body string Stripe sent. Parsing JSON, reading the request twice, changing encoding, adding whitespace, or using framework middleware that mutates the body can make a real Stripe event fail verification.

Next.js App Router pattern

export async function POST(req: Request) {
  const body = await req.text();
  const sig = req.headers.get('stripe-signature');

  const event = stripe.webhooks.constructEvent(
    body,
    sig!,
    process.env.STRIPE_WEBHOOK_SECRET!
  );

  return Response.json({ received: true });
}

Avoid these before verification

  • await req.json() before constructEvent.
  • Re-serializing the payload with JSON.stringify.
  • Shared API wrappers that parse every request body.
  • Reading the stream once for logging and again for Stripe.

3. Separate delivery status errors from signature errors

A 307, 308, 401, or 403 is not primarily a Stripe signature problem. It means Stripe may not be reaching your handler cleanly. Fix redirects, middleware, auth guards, and endpoint URL mismatch first. Then debug signature verification.

  • A 400 response with “missing signature” from a manual curl can prove the route is reachable.
  • A Stripe delivery attempt with stripe-signature present and handler logs around constructEvent points to secret/body issues.
  • A delivery attempt that never appears in Vercel function logs points to URL, domain, route, middleware, or deployment protection issues.

4. Check Vercel env targeting and redeploy behavior

Production-only failures often happen because local envs are correct while Vercel's production deployment has an old, missing, preview-only, or test-mode STRIPE_WEBHOOK_SECRET.

Production env checklist

  • STRIPE_WEBHOOK_SECRET exists in Vercel Production.
  • The value matches the Dashboard endpoint secret for that URL and mode.
  • Live secret is paired with live Stripe events; test secret with test events.
  • The app was redeployed after env changes.

Safe logging

// Do not log secrets.
console.log({
  hasWebhookSecret: Boolean(process.env.STRIPE_WEBHOOK_SECRET),
  secretPrefix: process.env.STRIPE_WEBHOOK_SECRET?.slice(0, 6),
  hasSignature: Boolean(req.headers.get('stripe-signature')),
});

5. Keep runtime and route shape boring

For a first working production webhook, choose the simplest route shape: a public POST route, Node-compatible Stripe SDK usage, raw body read once, signature verification first, idempotent database updates after verification, and a fast 2xx response.

  • If using Edge runtime, confirm your Stripe SDK/signature path is compatible. When unsure, force a standard Node.js runtime for the webhook route.
  • Do not put the webhook behind Supabase session checks, CSRF checks, route groups that redirect, or “API auth” wrappers.
  • Do not use the checkout success page as proof the webhook worked. Stripe delivery logs and app function logs are the source of truth.

What to send for a signature verification review

Do not send secrets. Send evidence that proves which of the three inputs is likely wrong: endpoint secret source, signature header presence, and raw-body route code.

Quick diagnosis table

Works locally, fails in production

Most likely CLI secret vs Dashboard secret, Vercel production env drift, or test/live mode mismatch.

Fails after req.json()

Read req.text() and pass that raw string directly to constructEvent.

Fails with 307/401 before handler logs

Fix delivery route status first: endpoint URL, redirects, middleware, auth guards, or deployment protection.

Started failing after env change

Check Vercel Production target and redeploy. Preview env values do not repair an existing production deployment.
