Stripe + Supabase renewal drift debug

Stripe invoice.paid double-extended Supabase membership?

A common billing drift bug is treating every invoice.paid as a fresh month to add. The first invoice, a retried invoice, a duplicate webhook delivery, or a manual repair can then extend Supabase access twice even though Stripe has one real billing period.

The fix is to make the invoice handler idempotent and copy Stripe's subscription period window instead of doing date arithmetic in your app.

The invariant to protect

One paid Stripe invoice should apply one local billing state transition. Membership should reflect Stripe's current period for the subscription, not the number of times a webhook handler ran.

1. Split first invoice from renewal invoice

invoice.paid can fire for the initial subscription invoice and later renewal invoices. Your app should decide what each case means instead of blindly adding time.

  • For initial checkout, create or activate the local subscription once.
  • For renewal invoices, update the existing subscription period from Stripe.
  • For retried invoices, avoid extending access again if the invoice was already applied.
  • For manual invoices or repairs, require an explicit mapping before changing membership.

If renewal invoices also mint credits, sessions, or usage allowance, check the monthly credit grant replay checklist before treating membership expiry as the only state to reconcile.

Useful Stripe fields to inspect include invoice.id, invoice.subscription, billing_reason, and the subscription period timestamps.

2. Make invoice handling idempotent by invoice ID

Store the Stripe invoice ID before or inside the same transaction that changes membership. If the same invoice arrives again, the handler should exit without changing dates.

-- Example shape only: adapt names to your app.
create table stripe_invoice_applications (
  stripe_invoice_id text primary key,
  stripe_subscription_id text not null,
  owner_id uuid not null,
  applied_at timestamptz not null default now(),
  applied_period_start timestamptz,
  applied_period_end timestamptz
);
  • Do not use only the Stripe event ID as the idempotency key; different events can describe the same invoice.
  • Do not add 30 days to the previous local expiry each time invoice.paid arrives.
  • Do not create a new local subscription row for a renewal invoice unless the Stripe subscription ID is genuinely new.

3. Copy Stripe period windows, not local guesses

Subscription periods can shift because of trials, coupons, proration, annual plans, failed payment recovery, or manual admin changes. Local membership should follow the Stripe subscription state.

// Shape, not drop-in code
const subscription = await stripe.subscriptions.retrieve(invoice.subscription);

await upsertLocalSubscription({
  stripeSubscriptionId: subscription.id,
  status: subscription.status,
  currentPeriodStart: new Date(subscription.current_period_start * 1000),
  currentPeriodEnd: new Date(subscription.current_period_end * 1000)
});

If your API version or webhook payload shape differs, retrieve the subscription directly rather than relying on a field that may be missing from the invoice object.

4. Regression tests worth keeping

  • The same invoice.paid delivered twice only applies once.
  • Initial invoice activation does not also count as a renewal extension.
  • A retried paid invoice does not add another month on top of the Stripe period.
  • Renewal changes the local period to Stripe's current period end.
  • An invoice for an unknown subscription is queued for repair instead of creating paid access for the wrong user.
