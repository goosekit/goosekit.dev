Stripe + Supabase duplicate subscription debug

Stripe created duplicate subscriptions and Supabase lost track?

Duplicate subscription drift happens when Stripe keeps billing one subscription while your app now points at another, or when canceling one subscription revokes too much access because Supabase cannot tell which service or workspace it belongs to.

This is not only a webhook idempotency bug. It is a source-of-truth bug across customer, subscription, service, workspace, and entitlement records.

The invariant to protect

One billing owner should have one deliberate subscription model. If multiple Stripe subscriptions are allowed, each subscription must map to one local service, workspace, entitlement set, or add-on scope. No active Stripe subscription should be invisible to Supabase.

1. Decide whether duplicates are forbidden or intentional

Before patching code, write the product rule. Many SaaS bugs come from accidentally mixing two models.

Single subscription model

One customer or workspace has one active subscription. Add-ons, seats, and usage should be subscription items, prices, quantities, or local entitlements under that subscription.

Multiple subscription model

Each subscription has a clear local owner and service scope. Canceling one subscription only revokes the matching service, workspace, seat pool, or entitlement group.

2. Find Stripe-active subscriptions that Supabase no longer tracks

The dangerous leak is an old subscription still billing after your app has overwritten or ignored its local record.

-- Example shape only: adapt table names to your app.
select
  bc.owner_id,
  bc.stripe_customer_id,
  s.stripe_subscription_id,
  s.status,
  s.service_key,
  s.updated_at
from billing_customers bc
join subscriptions s on s.owner_id = bc.owner_id
where s.status in ('active', 'trialing')
  and (
    s.service_key is null
    or not exists (
      select 1 from service_entitlements e
      where e.subscription_id = s.id
    )
  );
  • Compare the Stripe Dashboard active subscriptions for the customer with local subscription rows.
  • Flag active Stripe subscriptions with no local service or entitlement mapping.
  • Flag local premium access where the matching Stripe subscription is canceled, unpaid, or missing.
  • Keep Stripe test mode and live mode separated while reconciling.

3. Stop add-on checkout from minting a second hidden subscription

If the customer already has a subscription and the add-on should belong to it, do not create a separate Checkout Session that starts a new subscription unless the product model says so.

  • For add-ons on the same billing relationship, update subscription items or quantities instead of creating a second subscription.
  • If you do create a second subscription, store the local service key and owner before access is granted.
  • Use an idempotency key for the add-on purchase path, especially if a retry can run the same order twice.
  • Reject or recover from a second active subscription when your product rule says only one should exist.

4. Make cancellation revoke the exact scope, not everything

Customer Portal changes and manual cancellations should update only the entitlements tied to the Stripe subscription that changed.

// Shape, not drop-in code
const subscription = await findLocalSubscriptionByStripeId(event.data.object.id);

await updateSubscriptionStatus({
  subscriptionId: subscription.id,
  status: stripeStatus,
  currentPeriodEnd
});

await setEntitlementsForSubscription({
  subscriptionId: subscription.id,
  serviceKey: subscription.service_key,
  active: ['active', 'trialing'].includes(stripeStatus)
});
  • Never revoke all customer access just because one subscription canceled unless there is only one subscription by design.
  • Do not overwrite the stored subscription ID with a newer checkout before old billing is closed or migrated.
  • Keep a manual repair note for merged or migrated duplicate subscriptions.
  • Log the Stripe subscription ID on every access change caused by billing.

5. Regression tests worth keeping

  • Add-on retry does not create a second subscription when the model is single-subscription.
  • Two active subscriptions for one customer are either blocked or both mapped to explicit local service scopes.
  • Canceling one subscription revokes only that subscription's entitlements.
  • A legacy duplicate subscription is detected as untracked billing, not silently ignored.
  • Portal upgrade, downgrade, cancel-at-period-end, and invoice events cannot overwrite an unrelated subscription row.

What to send for a duplicate-subscription review

Send facts, not secrets: one Stripe customer ID, the active subscription IDs, the local owner row, the local subscription/entitlement rows, and the checkout or add-on path that created the duplicate.
