Stripe Connect + payout ledgers

Stripe Connect transfer.reversed but your clawback ledger drifted?

A Connect reversal is not just a Stripe webhook. It can change a payout record, an available balance row, a commission state, and a clawback or recovery queue. If one side effect is skipped or replayed with the wrong sign, your platform ledger stops matching provider truth.

The useful review is narrow: one transfer.reversed event should unwind the same local money state exactly once, even after duplicate delivery or a mid-handler failure.

The invariant to protect

One Stripe Connect transfer reversal should produce one local reversal result: payout record updated, payout balance adjusted with the correct sign, commission moved into the expected recovery state, and clawback queue created or confirmed without duplicate debit.

Fixed-scope review fit: usually EUR149 when Stripe Connect payout balances, commission state, clawback recovery, and retry/replay are all in scope.

1. Name each money-state row explicitly

Before testing idempotency, list every local row touched by a reversal. If the handler hides these updates behind one helper, failures become hard to reason about.

Stripe truth

  • Transfer id
  • transfer.reversed event id
  • Reversal amount and currency
  • Connected account id

App truth

  • Payout record row
  • Payout balance row
  • Commission or platform-fee row
  • Clawback queue or recovery row

2. Guard the sign and replay behavior together

Most expensive bugs are not missing webhooks. They are sign errors or duplicate local updates that only appear after a retry.

  • A reversal deducts available balance in one branch but credits it in another.
  • The payout record is marked reversed before the clawback row is durable.
  • An event id ledger skips a retry even though only half the local unwind completed.
  • Partial reversal and full reversal code paths use different signs or rounding.

3. Keep the handler atomic where money state splits

If your stack uses Supabase or Postgres, prefer one transactional boundary around payout record, payout balance, commission, and clawback queue writes. A SECURITY DEFINER RPC can be safer than four separate client calls.

-- Example shape only.
select process_transfer_reversal(
  provider_event_id,
  transfer_id,
  reversal_amount_cents,
  connected_account_id
);
  • Use the provider event id as a replay key, but do not mark it complete too early.
  • Lock the payout balance row before applying deltas.
  • Store the before/after balance or a signed ledger delta for auditability.
  • Return the existing reversal result on duplicate delivery.

4. Replay these cases before trusting the ledger

  • Known transfer reversed once.
  • Unknown or already-reconciled transfer.
  • Partial reversal after an earlier partial reversal.
  • Duplicate transfer.reversed event delivery.
  • Injected failure after payout record update but before clawback queue insert.
  • Replay after the injected failure returns one coherent local state.
