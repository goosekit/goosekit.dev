Next.js billing debug

Stripe webhook not updating Supabase paid status?

The awkward failure mode is common: checkout succeeds, Stripe shows a customer or subscription, but your Next.js app still treats the user as free. Use this checklist to find whether the bug is in the webhook, user mapping, database update, or route gate.

Start with the likely failure

If the user paid but the app did not unlock, do not keep changing the checkout button. Follow the path: checkout metadata → webhook signature → event handler → Supabase write → app-side paid-state read.

1. Confirm checkout is carrying the right app user

  • Pass a stable internal user ID in Stripe Checkout metadata, for example supabase_user_id.
  • Store or reuse the Stripe customer ID for that same internal user. Avoid creating a new customer on every checkout attempt.
  • Check the successful Checkout Session in Stripe Dashboard and confirm the metadata is present there, not only in your code.
  • Make the success URL cosmetic only. Unlocking paid access should come from webhook-confirmed state, not from visiting /success. If that path is already writing access, use the Checkout success URL replay checklist.

2. Verify the webhook signature with the raw body

In Next.js, many webhook bugs come from parsing the request before Stripe verifies the signature. Stripe needs the exact raw request body plus the correct signing secret for that endpoint.

  • Use the webhook signing secret for the same endpoint and mode: local CLI, test dashboard, and live dashboard are different.
  • Do not verify against already-parsed JSON.
  • Log rejected signature errors separately from handler errors so you know whether the event reached your business logic.
  • In production, confirm Stripe is posting to your real domain, not a preview or old Vercel URL.

3. Handle the subscription events that change access

For a subscription SaaS, checkout.session.completed is not enough. It starts the story; later events keep the app state honest.

Minimum events

  • checkout.session.completed
  • customer.subscription.updated
  • customer.subscription.deleted
  • invoice.paid
  • invoice.payment_failed

State to store

  • Stripe customer ID
  • Subscription ID
  • Price/plan ID
  • Status: trialing, active, past_due, canceled
  • Last webhook event ID and timestamp

4. Make Supabase writes idempotent and server-only

  • Use a server-side Supabase client with the correct service role only inside the webhook route or trusted backend code.
  • Upsert by a stable unique key such as user ID, customer ID, or subscription ID.
  • Record processed Stripe event IDs so duplicate webhook delivery does not create inconsistent state.
  • Fail closed: if the update fails, log it and keep the user unpaid until a real webhook/database write succeeds.

5. Test the app-side gate separately

Sometimes Stripe and Supabase are correct, but the dashboard still reads stale state or checks the wrong table.

  • Inspect the exact Supabase row after a test payment. Confirm it has the expected user ID and paid status.
  • Test with a brand-new browser profile and a brand-new user, not only your admin account.
  • Make the protected route read from your internal paid-state table, not directly from Stripe on every request.
  • Add a small internal debug view before launch: user ID, customer ID, subscription status, last webhook time.

6. If paid access starts jobs, test the launch gate too

Some products do not just unlock a dashboard. They start paid work: AI runs, automations, exports, campaign generation, or background jobs. In that case the billing gate must check more than a stale local active flag.

  • A terminal Stripe status such as canceled or a deleted subscription must beat any old active Supabase row.
  • invoice.paid should not accidentally revive a scheduled cancellation or bypass a newer terminal event.
  • The job launch path should check subscription state, tier access, monthly run cap, and credit reservation before work starts.
  • If credit reservation fails after access passes, the job should not launch without a repairable reservation row.

When to stop debugging alone

If checkout succeeds but access still fails after checking metadata, signature verification, webhook events, Supabase writes, and route gating, the next useful step is a narrow repo review — not another generic tutorial.
