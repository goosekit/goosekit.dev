What gets checked

Provider truth

  • Checkout Session, PaymentIntent, customer, subscription, invoice, refund, portal, trial, pause, and dunning states relevant to the app.
  • Webhook delivery status, retries, duplicate events, missing handlers, and signature/environment drift.
  • Which Stripe object is the source of truth for the access decision.

App truth

  • Supabase rows that store plan, status, workspace, credits, seats, entitlements, or booking/payment state.
  • Access-gate behavior for paid, unpaid, canceled, refunded, paused, and recovered customers.
  • Reconciliation query or manual check that can find mismatches before users report them.

Send a small evidence packet first

A good audit request does not need production secrets or a full repo dump. It needs one concrete mismatch and enough sanitized state to decide whether the fixed pass is safe to scope.

Minimum packet

  • One Stripe object id: Checkout Session, subscription, invoice, customer, PaymentIntent, refund, or event id.
  • The Supabase table and row that should represent the same paid, unpaid, refunded, or canceled state.
  • The webhook route and recent delivery status: delivered, failed, retried, duplicated, or not configured.
  • Expected app state and actual app state, stated plainly enough to reproduce the mismatch.
  • The user-visible symptom: paid but blocked, unpaid but unlocked, duplicate access, wrong workspace, or stale credits/seats.

Do not send yet

  • API keys, webhook signing secrets, service-role keys, customer card data, or production database credentials.
  • Long source archives before the mismatch is scoped.
  • A vague "review my billing" request without an expected app state.
  • Private user data that is not needed to identify the billing state transition.

If you can run one SQL query first

The fastest audit starts with a reconciliation row: Stripe status, local access status, customer id, user or workspace id, and the reason they disagree. If you do not already have that query, generate a skeleton before sending the packet.

Useful rows

  • Stripe active or trialing, but Supabase is free, missing, or inactive.
  • Stripe canceled, unpaid, incomplete, refunded, or past_due, but Supabase is still paid.
  • Stripe customer or subscription row exists without a matching local user, team, or workspace.

Audit impact

  • Rows found: the audit can start from concrete revenue leakage or customer blocking.
  • No rows found: the likely scope shifts to webhook coverage, idempotency, and recurring monitoring.
  • Query cannot be written: the first problem is usually schema clarity, not billing logic.
