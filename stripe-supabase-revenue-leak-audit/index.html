What gets checked

Provider truth

  • Checkout Session, PaymentIntent, customer, subscription, invoice, refund, portal, trial, pause, and dunning states relevant to the app.
  • Webhook delivery status, retries, duplicate events, missing handlers, and signature/environment drift.
  • Which Stripe object is the source of truth for the access decision.

App truth

  • Supabase rows that store plan, status, workspace, credits, seats, entitlements, or booking/payment state.
  • Access-gate behavior for paid, unpaid, canceled, refunded, paused, and recovered customers.
  • Reconciliation query or manual check that can find mismatches before users report them.

Good fit

  • Live or near-live SaaS with Stripe payments, subscriptions, credits, deposits, bookings, or seat limits.
  • Next.js + Supabase + Stripe, or a very close stack with the same billing state problem.
  • You can share sanitized object ids, table names, logs, screenshots, and the expected paid-access policy.
  • You want a bounded audit, not unlimited implementation.

Not a fit

  • No Stripe billing surface exists yet.
  • The request is “build my whole subscription system”.
  • The issue requires private production access before basic evidence is shared.
  • The business cannot name what paid, unpaid, canceled, or refunded should mean in the app.

Audit output

The output is designed to be useful even if no implementation is included: exact mismatch cases, suspected root cause, the smallest reliable test/fix plan, and whether the issue is a one-time setup bug or a recurring billing-health risk.
